# OpenClaw x 飞书机器人多赛事管理系统：开发任务拆解清单 v1

## 0. 文档目标
把当前已有的 3 份设计文档继续下沉为可执行开发任务，便于：
- 直接排期
- 分模块开发
- 明确依赖关系
- 控制首版范围

当前已具备基础文档：
1. `PRD-v1.1.md`
2. `数据设计文档-v1.md`
3. `事件与指令设计文档-v1.md`

本清单按 **P0 / P1 / P2** 拆分，并尽量遵循“先跑通主链路，再补增强项”的顺序。

---

## 1. 目标交付物
首版目标不是“一次做满”，而是先完成一个稳定可跑的闭环：

### 首版闭环
1. 多赛事映射可用
2. 管理员白名单可用
3. `/检查注册` 可用
4. `/暂停顺延` / `/恢复顺延` 可用
5. `/手动签到 [场次]` 可用
6. 签到卡片仅双方队长可点
7. 比分达到赛点后可自动顺延下一场签到
8. 战报可发送（至少文本版）
9. 审计日志覆盖关键动作

1. **配置与赛事映射模块**
2. **Bitable 数据访问模块**
3. **管理员指令模块**
4. **签到卡片模块**
5. **比分监听与顺延模块**
6. **战报发送模块**
7. **审计日志模块**
8. **测试与联调模块**

## 3. P0：必须先做的任务

## 模块 A：配置与赛事映射模块（P0）
### 目标
让系统能先正确识别“当前是哪个赛事”，这是所有逻辑入口。

### 任务 A1：读取赛事配置表
- 能从 Bitable 读取《赛事配置表》
- 拉取字段：TournamentID、ChatID、赛事状态、顺延开关、BitableAppToken
- 输出统一结构对象

### 任务 A2：建立 ChatID → TournamentID 映射
- 启动时初始化缓存
- 支持按 ChatID 查到当前赛事配置
- 查不到时直接中断后续逻辑

### 任务 A3：赛事状态基础校验
- 若赛事状态不是“进行中 / 筹备中”等允许状态，则部分逻辑不执行
- 顺延类逻辑额外校验 `顺延开关`

### 产出
- `getTournamentContextByChatId(chatId)`
- `getTournamentContextByTournamentId(tournamentId)`

---

## 模块 B：Bitable 数据访问模块（P0）
### 目标
统一所有表的读写入口，避免后面逻辑层到处散写 API。

### 任务 B1：封装赛事配置表查询
### 任务 B2：封装选手表查询
### 任务 B3：封装战队表查询
### 任务 B4：封装场次表查询/更新
### 任务 B5：封装卡组表查询/更新
### 任务 B6：封装白名单表查询
### 任务 B7：封装审计日志写入

### 要求
- 所有查询默认带 `TournamentID` 过滤
- 所有更新接口明确 record_id / fields 输入格式
- 同一表写入尽量串行，避免冲突

### 产出建议
- `bitable/tournament-config.ts`
- `bitable/player-master.ts`
- `bitable/team-master.ts`
- `bitable/match-results.ts`
- `bitable/deck-submission.ts`
- `bitable/admin-whitelist.ts`
- `bitable/audit-log.ts`

---

## 模块 C：管理员指令模块（P0）
### 目标
跑通群内管理员指令入口。

### 任务 C1：消息解析器
- 识别 `/检查注册`
- 识别 `/暂停顺延`
- 识别 `/恢复顺延`
- 识别 `/手动签到 [场次]`
- 识别 `/重发战报 [场次]`
- 识别 `/公示卡组`

### 任务 C2：白名单校验
- 根据 operatorOpenId + TournamentID 查询白名单
- 不通过时直接返回失败反馈

### 任务 C3：注册检查逻辑
- 读取选手表、战队表
- 汇总异常
- 按战队生成提醒消息

### 任务 C4：顺延暂停 / 恢复逻辑
- 更新赛事配置表中的 `顺延开关`
- 写审计日志

### 任务 C5：手动签到逻辑
- 支持按场次序号或 MatchID 定位场次
- 生成并发送签到卡片
- 必要时把场次状态改为 `待签到`

### 任务 C6：公示卡组逻辑（可放 P1，若时间紧）
- 校验可公示卡组
- 发送公示消息
- 更新卡组状态

---

## 模块 D：签到卡片模块（P0）
### 目标
跑通“卡片发出 → 队长点击 → 写回签到结果”的完整闭环。

### 任务 D1：设计签到卡片模板
卡片至少包含：
- 场次名称
- 对阵双方
- 两个签到按钮（A队 / B队）
- 提示文案

### 任务 D2：生成卡片 payload
按钮必须包含：
- action
- tournament_id
- match_id
- team_side
- version

### 任务 D3：卡片发送器
- 在群中发送卡片
- 同时 @ 双方队长
- 保留消息发送结果用于排查

### 任务 D4：卡片点击回调处理
- 解析 payload
- 校验点击人身份
- 更新 A队签到 / B队签到
- 更新签到状态
- 若双方已签，则更新 `MatchStatus = 已签到待开赛`

### 任务 D5：卡片幂等控制
- 已签到重复点击时不重复写入
- 非法点击不写库
- 所有点击写审计日志

---

## 模块 E：比分监听与顺延模块（P0）
### 目标
跑通“比分更新 → 达到赛点 → 自动发下一场签到”的核心链路。

### 任务 E1：接入 Bitable 记录更新事件
- 仅处理《赛事对阵与赛果表》记录更新
- 识别关键字段变化：A大分 / B大分 / MatchStatus

### 任务 E2：防抖处理
- 同一场次事件延迟 3 秒再回查当前记录
- 只处理最终稳定比分

### 任务 E3：顺延条件判断器
满足以下才返回 true：
- 顺延开关开启
- 当前场次状态为进行中
- 当前记录尚未触发顺延
- 首次达到赛点
- 下一场存在且状态允许

### 任务 E4：幂等控制
- 生成 `trigger_key`
- 处理前先查是否已执行
- 命中重复时写简日志并结束

### 任务 E5：顺延执行器
- 定位下一场
- 发送签到卡片
- 更新下一场状态
- 更新当前场次 `是否已触发顺延 = true`
- 记录顺延触发时间
- 写审计日志

### 任务 E6：暂停恢复边界处理
- 暂停时比分仍可更新，但不自动发卡
- 恢复后不追历史事件

---

## 模块 F：战报发送模块（P0/P1）
### 目标
场次结束后可稳定发战报，至少先保证文本版。

### 任务 F1：结束态识别
- 当场次状态变为 `已结束` 时进入战报逻辑

### 任务 F2：文本战报生成器（P0）
- 拼接场次、对阵双方、比分、小局明细
- 先保证文本版一定能发

### 任务 F3：战报海报生成器（P1）
- 后续可接图片模板
- 失败时自动回退文本版

### 任务 F4：重发战报逻辑
- 支持 `/重发战报 [场次]`
- 避免重复刷屏，可加“本次为重发”标记

---

## 模块 G：审计日志模块（P0）
### 目标
关键动作都有迹可循。

### 任务 G1：定义统一日志写入函数
输入至少包含：
- TournamentID
- 动作类型
- 操作者
- 操作来源
- 目标对象ID
- 成功/失败
- 错误原因 / 摘要

### 任务 G2：接入所有关键节点
至少覆盖：
- 管理员指令
- 卡片点击
- 顺延触发
- 战报发送
- 公示卡组
- 重复事件跳过

### 任务 G3：失败日志单独标记
便于后面筛选异常情况

---

## 模块 H：测试与联调模块（P0）
### 目标
在真实赛事上之前，先把关键链路测透。

### 任务 H1：测试赛事配置准备
- 测试群
- 测试赛事数据
- 测试战队
- 测试场次链路

### 任务 H2：管理员指令测试
- 白名单内执行成功
- 非白名单执行失败
- 参数缺失时有清晰反馈

### 任务 H3：签到卡片测试
- A 队长签到成功
- B 队长签到成功
- 普通队员点击失败
- 重复点击幂等生效

### 任务 H4：顺延测试
- 比分未达赛点时不发下一场
- 比分首次达到赛点时发下一场
- 重复更新比分不重复发卡
- 暂停顺延后不触发
- 恢复后新事件可继续触发

### 任务 H5：战报测试
- 已结束场次能发文本战报
- 海报失败时可降级
- `/重发战报` 可工作
