# OpenClaw x 飞书机器人多赛事管理系统：事件与指令设计文档 v1

## 0. 文档目标
本文件用于定义首版机器人交互协议与自动化事件处理逻辑，供开发直接实现。覆盖范围：
- 管理员指令
- 交互式签到卡片
- Bitable 变更事件
- 比分驱动顺延
- 文案反馈与错误处理
- 幂等、防抖、暂停恢复规则

本版默认沿用当前确认方案：
- **权限模型不扩展**
- 管理类操作由 **管理员白名单** 控制
- 签到按钮仅允许 **对应场次双方队长** 点击

---

## 1. 事件源总览

### 1.1 输入事件
系统首版主要处理以下两类输入：

1. **飞书消息事件**
   - 来源：`im.message.receive_v1`
   - 用途：识别管理员指令

2. **Bitable 记录更新事件**
   - 来源：`bitable.table_record_updated_v1`
   - 用途：监听比分变化、卡组审核变化等

3. **卡片按钮点击回调**
   - 来源：飞书交互卡片 action callback
   - 用途：处理签到按钮点击

---

## 2. 统一上下文字段
为了避免多赛事串号，所有处理逻辑都必须在运行时构造统一上下文对象。

```ts
context = {
  tournamentId,
  chatId,
  operatorOpenId,
  operatorName,
  sourceType,      // command | card_action | bitable_event
  targetTable,
  targetRecordId,
  messageId,
  eventId,
  timestamp
}
```

### 必做校验
所有逻辑执行前必须校验：
1. 当前 ChatID 能否映射到 TournamentID
2. 操作者是否属于当前赛事的合法上下文
3. 目标记录的 TournamentID 是否与当前上下文一致
4. 是否命中暂停 / 已处理 / 重复触发条件

## 3. 管理员指令设计

## 3.1 指令识别规则
管理员指令从群消息中识别，格式如下：

| 指令 | 参数 | 示例 |
|---|---|---|
| `/检查注册` | 无 | `/检查注册` |
| `/公示卡组` | 无 | `/公示卡组` |
| `/暂停顺延` | 无 | `/暂停顺延` |
| `/恢复顺延` | 无 | `/恢复顺延` |
| `/手动签到 [场次]` | 场次序号 / MatchID | `/手动签到 3` |
| `/重发战报 [场次]` | 场次序号 / MatchID | `/重发战报 M_R1_003` |

### 指令识别建议
- 去掉首尾空格再判断
- 支持全角空格容错
- `/手动签到`、`/重发战报` 要支持按“场次序号”或“MatchID”定位

---

## 3.2 白名单校验逻辑
任何管理员指令执行前，必须做以下判断：

1. 从当前群 `chatId` 找到 `TournamentID`
2. 到《管理员白名单》表中查询：
   - `TournamentID == 当前赛事`
   - `飞书UserID == operatorOpenId`
   - `是否启用 == true`
3. 若不满足，则拒绝执行

### 拒绝反馈
建议文案：
- `你不在本赛事管理员白名单内，无法执行该指令。`

并写入审计日志：
- 动作类型：管理员指令
- 处理结果：失败
- 错误原因：白名单校验失败

---

## 4. 指令详细设计

## 4.1 `/检查注册`

### 作用
扫描当前赛事群内成员与《选手主档案库》的匹配情况，识别异常并仅 @ 对应战队队长催办。

### 输入
- 指令发起者：管理员白名单用户
- 当前群 ChatID

### 执行流程
1. 校验管理员身份
2. 根据 ChatID 找到 TournamentID
3. 获取当前群成员列表（如果当前实现拿不到完整群成员，则用已有同步结果替代）
4. 读取《选手主档案库》与《战队主档案库》
5. 按战队汇总异常：
   - 队员未建档
   - 战网 ID 缺失
   - 定妆照缺失
   - 已报名但未入群
   - 战队人数不完整
6. 生成按战队分组的提醒文本
7. 在群内发提醒，**仅 @ 对应战队队长**
8. 写审计日志

### 输出
- 群内汇总提醒
- 不对普通队员单独点名

### 成功反馈建议
- `注册检查完成，已按战队汇总未完成项并提醒对应队长。`

### 无异常反馈建议
- `注册检查完成，当前未发现需补充的战队信息。`

---

## 4.2 `/公示卡组`

### 作用
在确认公示时机后，将卡组公示视图或公示链接发送到赛事群。

### 执行流程
1. 校验管理员身份
2. 找到 TournamentID
3. 校验赛事配置中的 `公示开关` 或强制执行策略
4. 检查是否存在“审核通过且当前有效”的卡组记录
5. 生成公示链接 / 公示视图地址
6. 在群内发送公示消息
7. 更新对应卡组记录：`是否已公示 = true`，必要时 `审核状态 = 已公示`
8. 写审计日志

### 失败场景
- 当前没有可公示卡组
- 仍有队伍处于待审核状态
- 链接或视图生成失败

### 反馈建议
- 成功：`卡组已公示。`
- 失败：`当前暂无可公示的卡组版本，请先确认审核状态。`

---

## 4.3 `/暂停顺延`

### 作用
挂起自动顺延逻辑。

### 执行流程
1. 校验管理员身份
2. 找到 TournamentID
3. 更新《赛事配置表》：`顺延开关 = false`
4. 写审计日志
5. 返回确认信息

### 成功反馈建议
- `已暂停本赛事自动顺延，后续比分变化将不再自动触发下一场签到。`

---

## 4.4 `/恢复顺延`

### 作用
重新启用自动顺延逻辑。

### 执行流程
1. 校验管理员身份
2. 找到 TournamentID
3. 更新《赛事配置表》：`顺延开关 = true`
4. 写审计日志
5. 返回确认信息

### 注意
- 恢复顺延后，仅对**后续新事件**生效
- 历史漏发场次不自动补发，由管理员使用 `/手动签到 [场次]` 处理

### 成功反馈建议
- `已恢复本赛事自动顺延。历史漏发场次如需补发，请使用 /手动签到。`

---

## 4.5 `/手动签到 [场次]`

### 作用
人工强制下发指定场次的签到卡片，用于首场开启、异常兜底或暂停恢复后的补发。

### 输入参数
- 场次序号 或 MatchID

### 执行流程
1. 校验管理员身份
2. 定位场次记录
3. 校验当前场次是否允许发卡：
   - MatchStatus 不为 `已结束`
   - 对阵双方存在
   - 队长信息存在
4. 生成签到卡片 payload
5. 在群里发送交互式卡片，并 **仅 @ 对应双方队长**
6. 若场次原状态为 `未开启`，可更新为 `待签到`
7. 写审计日志

### 成功反馈建议
- `已手动下发第 3 场签到卡片。`

### 失败反馈建议
- `未找到对应场次，请检查场次序号或 MatchID。`
- `该场次已结束，无法再次下发签到卡片。`

---

## 4.6 `/重发战报 [场次]`

### 作用
在战报生成失败或发送失败后手动重发指定场次战报。

### 执行流程
1. 校验管理员身份
2. 定位场次记录
3. 校验场次状态必须为 `已结束`
4. 读取比分信息
5. 优先尝试生成战报海报
6. 海报失败则降级发送文本战报
7. 更新 `战报发送状态`
8. 写审计日志

---

## 5. 交互式签到卡片设计

## 5.1 卡片发送时机
- 首场：管理员手动或定时开启
- 后续场次：顺延触发
- 异常兜底：管理员手动签到

## 5.2 卡片最小 payload 结构
建议按钮 value 中带以下字段：

```json
{
  "action": "match_checkin",
  "tournament_id": "T_HS_2026_S1",
  "match_id": "M_R1_003",
  "team_side": "A",
  "version": "v1"
}
```

B 队按钮则 `team_side = B`。

### 字段说明
- `action`：动作类型
- `tournament_id`：赛事隔离校验
- `match_id`：定位场次
- `team_side`：是 A 队签到还是 B 队签到
- `version`：卡片版本，便于未来兼容升级

---

## 5.3 卡片点击处理流程
1. 解析 payload
2. 校验 `action == match_checkin`
3. 根据 `tournament_id`、`match_id` 找到场次
4. 校验点击人是否为对应队伍的队长
5. 校验赛事未暂停、场次未结束
6. 校验该侧是否已签到
7. 写入签到结果：
   - A队按钮 → `A队签到 = true`
   - B队按钮 → `B队签到 = true`
8. 更新 `签到状态`
9. 若双方都签到，则：
   - `签到状态 = 已签到`
   - `MatchStatus = 已签到待开赛`
10. 写审计日志
11. 返回卡片点击反馈

---

## 5.4 卡片点击反馈文案

### 成功场景
- A 队刚签到成功：
  - `签到成功，已记录你方到场。等待对方队长签到。`
- 双方都签到完成：
  - `签到成功，双方已完成签到，本场可进入开赛状态。`

### 失败场景
- 非对应队长点击：
  - `你不是本场签到负责人，无法操作。`
- 已重复点击：
  - `你方已完成签到，无需重复操作。`
- 场次已结束：
  - `本场已结束，签到入口已关闭。`
- 赛事顺延暂停：
  - `当前赛事自动流程已暂停，请联系管理员处理。`

## 6. 比分更新事件设计

## 6.1 监听目标
监听《赛事对阵与赛果表》的记录更新，重点关注字段：
- `A大分`
- `B大分`
- `MatchStatus`
- `是否已触发顺延`

## 6.2 处理场景
### 场景 A：普通比分更新
- 仅更新场次比分，不触发顺延

### 场景 B：首次达到赛点
- 满足顺延条件时，下发下一场签到卡片

### 场景 C：比分定格且场次结束
- 更新 MatchStatus 为 `已结束`
- 触发战报发送逻辑

---

## 6.3 顺延触发判断逻辑
满足以下所有条件时，才允许触发自动顺延：

1. 当前赛事 `顺延开关 = true`
2. 当前场次 `MatchStatus = 进行中`
3. 当前记录 `是否已触发顺延 = false`
4. 当前比分**首次**达到赛点阈值
5. `下一场MatchID` 存在
6. 下一场存在有效记录
7. 下一场 `MatchStatus` 为 `未开启` 或 `待签到`
8. 未命中幂等重复检查

### 命中后动作
1. 生成下一场签到卡片
2. 群内发送卡片，并仅 @ 双方队长
3. 更新下一场：
   - 若原状态为 `未开启`，改为 `待签到`
4. 更新当前场次：
   - `是否已触发顺延 = true`
   - `顺延触发时间 = now`
5. 写审计日志

---

## 6.4 幂等策略
建议使用：

```text
trigger_key = tournament_id + current_match_id + next_match_id + trigger_type
```

例如：
`T_HS_2026_S1::M_R1_002::M_R1_003::reach_match_point`

### 规则
- 同一 `trigger_key` 若已成功处理，再次收到事件直接忽略
- 记录到日志时可标记为“重复事件已跳过”

---

## 6.5 防抖策略
为避免裁判短时间连续修改比分造成重复触发：
- 同一场次比分更新事件进入 3~5 秒防抖窗口
- 仅处理窗口结束后的最终状态

实现方式可选：
1. 内存定时器 + 最新事件覆盖
2. 简单延迟后回查 Bitable 当前记录

首版推荐：
- **延迟 3 秒后回查当前记录**
- 因为它比纯内存定时器更抗重启、也更稳

## 7. 战报发送设计

## 7.1 触发条件
- 场次状态更新为 `已结束`
- 且比分完整可读

## 7.2 流程
1. 定位场次与双方战队信息
2. 读取大分 / 小局明细
3. 尝试生成战报海报
4. 成功则发送海报
5. 失败则降级为文本战报
6. 更新 `战报发送状态`
7. 写审计日志

## 7.3 文本战报模板建议
```text
【赛果播报】
第 {场次序号} 场：{A队} vs {B队}
最终比分：{A大分} : {B大分}
小局记录：{小局明细}
```

## 8. 异常与兜底处理

## 8.1 非法点击签到
### 场景
- 队员点击
- 对方队长点错按钮
- 非本赛事人员点击

### 动作
- 不更新数据
- 返回失败提示
- 写审计日志（失败）

---

## 8.2 重复触发顺延
### 场景
- webhook 重复推送
- 比分连续保存多次
- 服务重启后重放事件

### 动作
- 依赖 `trigger_key` 与 `是否已触发顺延` 双保险拦截
- 写日志但不重复发卡

---

## 8.3 战报发送失败
### 动作
- 先降级文本战报
- 若文本也失败，则记录 `战报发送状态 = 失败`
- 允许管理员用 `/重发战报 [场次]` 补发

---

## 8.4 顺延暂停后的事件
### 规则
- 若 `顺延开关 = false`，比分事件仍可更新比分，但不触发下一场签到
- 恢复顺延后不追历史，仅继续处理后续新变化

---

## 9. 审计日志落点要求
以下动作必须写审计日志：
- 所有管理员指令
- 所有签到按钮点击（成功/失败都记）
- 每次自动顺延触发
- 每次战报发送
- 每次卡组公示
- 每次因幂等跳过的重复事件（可简写）

日志至少包含：
- TournamentID
- 动作类型
- 操作者
- 操作来源
- 目标对象ID
- 处理结果
- 错误原因 / 结果摘要
- 时间

---

## 10. 首版开发优先级建议

### P0（必须先做）
1. ChatID → TournamentID 映射
2. 管理员白名单校验
3. `/检查注册`
4. `/暂停顺延` / `/恢复顺延`
5. `/手动签到 [场次]`
6. 签到卡片点击链路
7. 比分监听 + 顺延触发幂等

### P1（紧随其后）
1. `/公示卡组`
2. 战报海报 / 文本降级发送
3. `/重发战报 [场次]`
4. 审计日志完善

### P2（可后续增强）
1. 定时首场签到
2. 更细的异常提醒模板
3. 更丰富的海报模板
4. 更完整的小局结构化数据
