# OpenClaw x 飞书机器人多赛事管理系统：数据设计文档 v1

## 0. 文档目标
本文件用于把 PRD v1.1 落成可执行的数据模型，供以下环节直接使用：
- 飞书多维表格（Bitable）建模
- 机器人读取 / 写入逻辑开发
- 管理员与运营后台操作规范
- 后续事件与指令设计文档引用

设计原则：
1. **多赛事隔离**：所有核心表必须带 `TournamentID`
2. **结构化优先**：能拆成独立字段，就不只存拼接文本
3. **人工与机器人边界清晰**：哪些字段人工维护，哪些字段机器人写入要明确
4. **可追溯**：关键动作保留日志与状态
5. **尽量少表但不牺牲可维护性**：先满足 2-3 个赛事并发运行

---

## 1. 总体建模方案

建议至少使用以下 7 张表：

1. **赛事配置表**（Tournament Config）
2. **选手主档案库**（Player Master）
3. **战队主档案库**（Team Master）
4. **赛事对阵与赛果表**（Match & Results）
5. **卡组提交与审核表**（Deck Submission）
6. **管理员白名单**（Admin Whitelist）
7. **审计日志表**（Audit Log）

---

## 2. 表关系总览

```text
赛事配置表(Tournament Config)
  ├── 1:N 选手主档案库(Player Master)
  ├── 1:N 战队主档案库(Team Master)
  ├── 1:N 赛事对阵与赛果表(Match & Results)
  ├── 1:N 卡组提交与审核表(Deck Submission)
  ├── 1:N 管理员白名单(Admin Whitelist)
  └── 1:N 审计日志表(Audit Log)

战队主档案库(Team Master)
  ├── 1:N 选手主档案库(Player Master)
  ├── 1:N 卡组提交与审核表(Deck Submission)
  └── 1:N 对阵表中的 A队 / B队

赛事对阵与赛果表(Match & Results)
  └── 自关联：下一场 MatchID
```

---

## 3. 字段类型约定（Bitable）

| 类型名称 | Bitable 类型建议 | 用途 |
|---|---|---|
| 唯一 ID | 文本 | 如 TournamentID / MatchID / TeamID |
| 人员 | 人员 | 队长、裁判、提交人 |
| 状态 | 单选 | 状态机字段 |
| 多标签异常 | 多选 | 异常标记 |
| 时间 | 日期时间 | 提交时间、更新时间、触发时间 |
| 布尔 | 复选框 | 是否已触发、是否在群 |
| 附件 | 附件 | 定妆照、卡组截图 |
| 长描述 | 多行文本 | 备注、驳回原因、日志摘要 |
| 关联 | 关联记录 | 战队 ↔ 选手 / 场次 ↔ 战队 |
| 查找引用 | 查找引用 | 从关联表回填队长、状态等 |
| 公式 | 公式 | 自动判定完整度、人数、签到状态等 |

## 4. 表一：赛事配置表（Tournament Config）

### 4.1 作用
作为整套系统的赛事主入口，负责保存赛事级配置、群映射和机器人开关。

### 4.2 字段设计

| 字段名 | 类型 | 必填 | 示例 | 维护方 | 说明 |
|---|---|---:|---|---|---|
| TournamentID | 文本 | 是 | T_HS_2026_S1 | 人工 | 赛事唯一主键 |
| 赛事名称 | 文本 | 是 | 炉石春季团体赛 | 人工 | 展示字段 |
| ChatID | 文本 | 是 | oc_xxx | 人工 | 飞书群 ID |
| BitableAppToken | 文本 | 是 | bascnxxx | 人工 | 当前赛事 Bitable 应用标识 |
| 赛事状态 | 单选 | 是 | 筹备中/进行中/已结束/已暂停 | 人工 | 赛事全局状态 |
| 顺延开关 | 复选框 | 是 | true | 人工/机器人 | 全局暂停/恢复顺延 |
| 公示开关 | 复选框 | 是 | false | 人工 | 是否允许卡组公示 |
| 首场签到开启方式 | 单选 | 否 | 手动/定时 | 人工 | 首场控制策略 |
| 默认签到超时分钟数 | 数字 | 否 | 10 | 人工 | 机器人提醒使用 |
| 赛事备注 | 多行文本 | 否 | - | 人工 | 补充说明 |
| 创建时间 | 创建时间 | 自动 | - | 系统 | 自动字段 |
| 更新时间 | 修改时间 | 自动 | - | 系统 | 自动字段 |

### 4.3 机器人使用方式
- 读：按 ChatID 找到 TournamentID 与开关配置
- 写：一般不主动写，只有少量开关同步场景可写

### 4.4 备注
这是一个非常关键的“入口表”。建议后端服务启动时优先把这张表加载成内存映射缓存。

| 字段名 | 类型 | 必填 | 示例 | 维护方 | 说明 |

| TournamentID | 文本 | 是 | T_HS_2026_S1 | 机器人/人工 | 赛事隔离键 |
| 飞书 UserID | 文本 | 是 | ou_xxx | 机器人/人工 | 绑定主键 |
| 飞书昵称快照 | 文本 | 否 | 张三 | 机器人 | 同步群成员昵称 |
| 报名姓名 | 文本 | 否 | 张三 | 人工/表单 | 报名用名 |
| 战网 ID | 文本 | 是 | player#1234 | 人工/表单 | 核心资料 |
| 定妆照 | 附件 | 是 | 1张图片 | 人工/表单 | 强制要求 |
| 所属战队 | 关联记录 | 否 | Team Master | 人工/机器人 | 关联战队 |
| 队内角色 | 单选 | 否 | 队长/正选/替补 | 人工 | 用于后续判断 |
| 是否在群 | 复选框 | 否 | true | 机器人 | 群成员同步结果 |
| 首次入群时间 | 日期时间 | 否 | 2026-04-03 20:00 | 机器人 | 首次同步写入 |
| 最后同步时间 | 日期时间 | 否 | 2026-04-03 21:00 | 机器人 | 最近一次扫描 |
| 档案状态 | 单选 | 是 | 未注册/待补资料/已完成/已锁定 | 机器人/人工 | 状态机字段 |
| 异常标记 | 多选 | 否 | 未入群/未绑定 | 机器人 | 支持多个异常 |
| 备注 | 多行文本 | 否 | - | 人工 | 额外说明 |
| 创建时间 | 创建时间 | 自动 | - | 系统 | 自动字段 |
| 更新时间 | 修改时间 | 自动 | - | 系统 | 自动字段 |

### 5.3 建议公式/派生逻辑
- **资料完整度**：若战网 ID 非空且定妆照非空 → 完整，否则不完整
- **档案状态建议规则**：
  - 未建记录：未注册
  - 已建记录但资料缺失：待补资料
  - 资料齐全：已完成
  - 赛事开始后：已锁定（如需要）

### 5.4 机器人读写边界
- **机器人读取**：注册检查、催缴、战队成员完整度判断
- **机器人写入**：是否在群、首次入群时间、最后同步时间、异常标记、部分状态
- **人工修改**：报名姓名、战网 ID、定妆照、所属战队、队内角色

## 7. 表四：赛事对阵与赛果表（Match & Results）

### 7.1 作用
这是整套系统最核心的一张表，签到、顺延、赛果、公示都围绕它转。

### 7.2 字段设计

| 字段名 | 类型 | 必填 | 示例 | 维护方 | 说明 |
|---|---|---:|---|---|---|
| MatchID | 文本 | 是 | M_R1_001 | 人工/机器人 | 唯一 ID |
| TournamentID | 文本 | 是 | T_HS_2026_S1 | 人工/机器人 | 赛事隔离键 |
| 场次序号 | 数字 | 是 | 1 | 人工 | 顺序控制 |
| 轮次 | 文本/单选 | 是 | 小组赛A组 | 人工 | 阶段信息 |
| BO类型 | 单选 | 是 | BO3 / BO5 | 人工 | 赛制 |
| 赛点阈值 | 数字 | 是 | 2 | 人工/公式 | BO5=2 |
| 对阵A队 | 关联记录 | 是 | Team A | 人工 | 关联战队 |
| 对阵B队 | 关联记录 | 是 | Team B | 人工 | 关联战队 |
| A队长 | 查找引用 | 否 | ou_xxx | 系统 | 从战队表取 |
| B队长 | 查找引用 | 否 | ou_xxx | 系统 | 从战队表取 |
| A队签到 | 复选框 | 否 | true | 机器人 | A队长点击后更新 |
| B队签到 | 复选框 | 否 | true | 机器人 | B队长点击后更新 |
| 签到状态 | 单选/公式 | 否 | 未签到/部分签到/已签到 | 系统/机器人 | 派生状态 |
| MatchStatus | 单选 | 是 | 未开启/待签到/已签到待开赛/进行中/已结束/已挂起 | 机器人/人工 | 场次状态机 |
| A大分 | 数字 | 否 | 2 | 裁判/机器人 | 结构化比分 |
| B大分 | 数字 | 否 | 1 | 裁判/机器人 | 结构化比分 |
| 小局明细 | 多行文本 | 否 | 1-0,1-1,2-1 | 裁判 | 可先简化存文本 |
| 是否已达赛点 | 公式/复选框 | 否 | true | 系统/机器人 | 顺延判断用 |
| 下一场MatchID | 文本 | 否 | M_R1_002 | 人工 | 顺延目标 |
| 是否已触发顺延 | 复选框 | 否 | false | 机器人 | 幂等关键字段 |
| 顺延触发时间 | 日期时间 | 否 | 2026-04-03 22:05 | 机器人 | 追踪触发时刻 |
| 裁判记录人 | 人员 | 否 | ou_referee | 裁判 | 比分责任人 |
| 比分最后更新时间 | 日期时间 | 否 | 2026-04-03 22:04 | 机器人/系统 | 更新痕迹 |
| 战报发送状态 | 单选 | 否 | 未发送/已发送/失败 | 机器人 | 公示兜底 |
| 备注 | 多行文本 | 否 | - | 人工 | 异常说明 |
| 创建时间 | 创建时间 | 自动 | - | 系统 | 自动字段 |
| 更新时间 | 修改时间 | 自动 | - | 系统 | 自动字段 |

### 7.3 关键公式/业务规则
- **签到状态**：
  - A、B 都未签到 → 未签到
  - 任一方签到 → 部分签到
  - 双方都签到 → 已签到
- **是否已达赛点**：
  - 若 `max(A大分, B大分) >= 赛点阈值`，则为 true
- **顺延触发限制**：
  - 只有 `是否已触发顺延 = false` 时才允许自动触发

### 7.4 机器人读写边界
- **机器人读取**：签到发卡、比分监听、顺延判断、战报发送
- **机器人写入**：A/B队签到、签到状态、MatchStatus、是否已触发顺延、顺延触发时间、战报发送状态
- **人工修改**：轮次、对阵关系、BO类型、下一场MatchID、异常备注
- **裁判修改**：比分、小局明细、裁判记录人

### 7.5 强提醒
不要把比分只存为 `2:1` 字符串。至少要存 `A大分`、`B大分` 两个数值字段。

| 提交时间 | 日期时间 | 是 | 2026-04-03 19:00 | 机器人 | 记录生成时间 |
| 版本号 | 数字 | 是 | 1 | 机器人 | 第几次提交 |
| 卡组代码集合 | 多行文本 | 是 | 多行 deckstring | 人工/机器人 | 存原始代码 |

| 驳回原因 | 多行文本 | 否 | 代码与截图不一致 | 人工 | 便于回看 |
| 是否当前有效版本 | 复选框 | 否 | true | 机器人/人工 | 当前使用版本 |
| 是否已公示 | 复选框 | 否 | false | 机器人/人工 | 公示动作状态 |
| 备注 | 多行文本 | 否 | - | 人工 | 其他情况 |
| 创建时间 | 创建时间 | 自动 | - | 系统 | 自动字段 |
| 更新时间 | 修改时间 | 自动 | - | 系统 | 自动字段 |

### 8.3 关键设计建议
- 每次重提都新增记录，不覆盖旧记录
- 同一战队在同一赛事下，只允许一个“当前有效版本”
- `审核状态` 和 `是否当前有效版本` 不要混用，两个字段分开更清楚

### 8.4 机器人读写边界
- **机器人读取**：判断是否已提交、是否已通过、是否允许公示
- **机器人写入**：SubmissionID、提交时间、版本号、Deckstring校验结果、是否已公示等
- **人工修改**：附件、审核状态、审核人、驳回原因、有效版本标记

| 处理结果 | 单选 | 是 | 成功/失败 | 机器人 | 结果 |

| 发生时间 | 日期时间 | 是 | 2026-04-03 22:05 | 机器人 | 记录时间 |

### 10.3 机器人读写边界
- **机器人读取**：通常只用于排查
- **机器人写入**：关键动作都应该写日志
- **人工修改**：不建议改；若改，需谨慎

## 11. 建表顺序建议
为了减少关联表来回返工，建议按这个顺序建：

1. 赛事配置表
2. 战队主档案库

## 13. 首版实现时可先简化的地方
为了尽快上线，首版可以先做这些简化：

1. **小局明细先用文本**
   - 不急着拆成更复杂子表
2. **赛事配置表先只保留一个 ChatID**
   - 后续若有公示群/裁判群再扩展多群映射
3. **卡组代码集合先存长文本**
   - 后期再考虑拆职业维度
4. **审计日志先记录关键动作**
   - 不必一开始就覆盖所有细枝末节
