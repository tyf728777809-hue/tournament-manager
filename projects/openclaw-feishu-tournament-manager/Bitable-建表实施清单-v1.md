# OpenClaw x 飞书机器人多赛事管理系统：Bitable 建表实施清单 v1

## 0. 文档目标
把前面的数据设计文档进一步落成“可实际操作的建表步骤”，用于：
- 在飞书多维表格里逐步建表
- 控制建表顺序，减少返工
- 明确哪些字段第一轮先建，哪些第二轮补公式 / 查找引用
- 每张表建完后立刻做最小验收

**原则：**
1. 先建“骨架字段”，后补公式和查找引用
2. 先建被依赖的表，再建关联表
3. 先保证能录数据，再追求自动化美观
4. 每建完一张表就做验收，不堆到最后一起测

---

## 1. 建表总顺序
推荐顺序如下：

1. **赛事配置表**
2. **战队主档案库**
3. **选手主档案库**
4. **赛事对阵与赛果表**
5. **卡组提交与审核表**
6. **管理员白名单**
7. **审计日志表**

### 为什么这么排
- 赛事配置表最先建，是全局入口
- 战队表先于选手表，方便后面选手关联战队
- 场次表依赖战队表
- 卡组表依赖战队表
- 白名单和日志表相对独立，后建也不影响主流程

## 2. 建表前准备

## 2.1 建议先新建一个独立 Bitable App
建议名称：
- `OpenClaw-多赛事管理系统-测试`
- 正式环境再复制为生产版

## 2.2 首轮只做“测试环境”
不要一上来就在正式赛事里搭。先做：
- 测试赛事
- 测试群
- 测试队伍
- 测试场次

## 2.3 字段命名要一次尽量定稳
建议：
- 中文展示字段保留
- 关键字段名称尽量不要频繁改
- 尤其是：`TournamentID`、`MatchID`、`TeamID`、`PlayerID`、`SubmissionID`

因为后面机器人写表时，字段名变来变去会很痛。

## 10. 第一轮建表后要立刻补的关联与公式
在 7 张表第一轮建完、并完成最小验收之后，再补第二轮自动化字段。

## 10.1 优先补的关联
1. 选手表 → 所属战队
2. 战队表 → 成员列表
3. 场次表 → 对阵A队 / 对阵B队
4. 卡组表 → TeamID

## 10.2 优先补的查找引用
1. 场次表里的 A队长 / B队长
2. 战队表里的成员同步摘要

## 10.3 优先补的公式
1. 战队表：正选人数 / 替补人数 / 已完成档案人数
2. 场次表：签到状态 / 是否已达赛点
3. 选手表：可选补资料完整度辅助字段

## 12. 实操建议：首版不要一开始就追求最优建模
我建议你实际建表时遵循这条原则：

> **先让表能支持机器人跑通主流程，再慢慢补“优雅设计”。**

具体体现为：
- `下一场MatchID` 先文本，不急着自关联
- `小局明细` 先文本，不急着拆结构
- `成员同步摘要` 先不做很复杂的展示
- 先保证机器人读写核心字段不受影响

这样速度会快很多，也更不容易卡在飞书字段细节上。

## 13. 建表阶段最容易踩的坑

### 坑 1：一开始就加太多公式
后果：
- 难排错
- 字段依赖混乱
- 修改表结构特别痛

### 坑 2：太早做复杂关联
后果：
- 字段联动异常时不容易定位
- 批量导入测试数据很痛苦

### 坑 3：关键 ID 字段没定稳
后果：
- 后面机器人代码全部跟着返工

### 坑 4：生产环境直接开工
后果：
- 测试数据污染正式赛事
- 错误提醒直接打到真实群
