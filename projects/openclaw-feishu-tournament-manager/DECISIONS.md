# 决策与变更记录 - OpenClaw x 飞书机器人多赛事管理系统

> 目的：不依赖会话上下文记忆。凡是方案调整、踩坑、取舍原因、临时结论，都在这里留下可追溯记录。

## 记录规则
- **记录什么**：关键决策、方案切换、遇到的问题、为何放弃某方案、当前临时结论
- **什么时候记**：每次出现“改方案 / 踩坑 / 新约束 / 新结论”时同步更新
- **怎么用**：新会话继续推进前，先看本文件 + `PROGRESS.md`

## 2026-04-02

### D-001：项目采用独立工作区管理
- **结论**：项目使用独立目录 `projects/openclaw-feishu-tournament-manager/`
- **原因**：避免和其他炉石/飞书项目混杂，方便持续沉淀设计与开发资产
- **影响**：后续 PRD、数据设计、事件协议、任务拆解统一收口在该目录下

### D-002：先补强 PRD，再进入设计与开发
- **结论**：基于初版 PRD 先整理出 `PRD-v1.1.md`
- **原因**：先把核心状态机、数据结构、顺延机制讲清楚，能减少后期返工
- **影响**：开发前先以 v1.1 作为统一底稿

### D-003：本轮不扩展权限模型
- **结论**：权限模型暂不细分，先沿用“管理员白名单 + 队长点击限制”
- **原因**：用户明确要求先按当前设定推进，避免前期设计发散
- **影响**：本轮文档与实现不引入更复杂的 RBAC

### D-004：不依赖聊天上下文记忆推进项目
- **结论**：后续凡是遇到方案切换、技术问题、实现取舍，都要同步写入项目文档
- **原因**：避免会话刷新、上下文切换后遗失关键脉络
- **影响**：`PROGRESS.md` 记录整体进度，`DECISIONS.md` 记录方案演进与踩坑

### D-005：首版数据模型采用 7 张核心表
- **结论**：首版采用 7 张表：赛事配置表、选手主档案库、战队主档案库、赛事对阵与赛果表、卡组提交与审核表、管理员白名单、审计日志表
- **原因**：在不把模型做得过重的前提下，已经能覆盖多赛事隔离、注册检查、签到顺延、卡组审核与全链路留痕
- **影响**：后续事件设计与建表实施以这 7 张表为准推进

### D-006：首版事件设计以“群内管理员指令 + 卡片签到 + Bitable比分监听”为主干
- **结论**：首版交互主干确定为三类输入：管理员指令、签到卡片回调、Bitable 记录更新事件
- **原因**：这样可以最小闭环覆盖注册检查、顺延签到、比分驱动和战报公示，不需要一开始就引入更复杂交互层
- **影响**：开发任务拆解将围绕这三类输入展开，优先实现主流程可跑通
