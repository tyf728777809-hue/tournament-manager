# 决策与变更记录 - OpenClaw x 飞书机器人多赛事管理系统

> 目的：不依赖会话上下文记忆。凡是方案调整、踩坑、取舍原因、临时结论，都在这里留下可追溯记录。

## 记录规则
- **记录什么**：关键决策、方案切换、遇到的问题、为何放弃某方案、当前临时结论
- **什么时候记**：每次出现“改方案 / 踩坑 / 新约束 / 新结论”时同步更新
- **怎么用**：新会话继续推进前，先看本文件 + `PROGRESS.md`

## 2026-04-02

### D-001：项目采用独立工作区管理
- **结论**：项目使用独立目录 `projects/openclaw-feishu-tournament-manager/`
- **原因**：避免和其他炉石/飞书项目混杂，方便持续沉淀设计与开发资产
- **影响**：后续 PRD、数据设计、事件协议、任务拆解统一收口在该目录下

### D-002：先补强 PRD，再进入设计与开发
- **结论**：基于初版 PRD 先整理出 `PRD-v1.1.md`
- **原因**：先把核心状态机、数据结构、顺延机制讲清楚，能减少后期返工
- **影响**：开发前先以 v1.1 作为统一底稿

### D-003：本轮不扩展权限模型
- **结论**：权限模型暂不细分，先沿用“管理员白名单 + 队长点击限制”
- **原因**：用户明确要求先按当前设定推进，避免前期设计发散
- **影响**：本轮文档与实现不引入更复杂的 RBAC

### D-004：不依赖聊天上下文记忆推进项目
- **结论**：后续凡是遇到方案切换、技术问题、实现取舍，都要同步写入项目文档
- **原因**：避免会话刷新、上下文切换后遗失关键脉络
- **影响**：`PROGRESS.md` 记录整体进度，`DECISIONS.md` 记录方案演进与踩坑

### D-005：首版数据模型采用 7 张核心表
- **结论**：首版采用 7 张表：赛事配置表、选手主档案库、战队主档案库、赛事对阵与赛果表、卡组提交与审核表、管理员白名单、审计日志表
- **原因**：在不把模型做得过重的前提下，已经能覆盖多赛事隔离、注册检查、签到顺延、卡组审核与全链路留痕
- **影响**：后续事件设计与建表实施以这 7 张表为准推进

### D-006：首版事件设计以“群内管理员指令 + 卡片签到 + Bitable比分监听”为主干
- **结论**：首版交互主干确定为三类输入：管理员指令、签到卡片回调、Bitable 记录更新事件
- **原因**：这样可以最小闭环覆盖注册检查、顺延签到、比分驱动和战报公示，不需要一开始就引入更复杂交互层
- **影响**：开发任务拆解将围绕这三类输入展开，优先实现主流程可跑通

### D-007：开发拆解按“先人工可控，再自动化增强”推进
- **结论**：首版开发顺序确定为：赛事映射 → 数据访问 → 管理员指令 → 签到卡片 → 比分顺延 → 战报发送 → 联调测试
- **原因**：这样可以先让主流程可控可测，再逐步引入自动顺延等高风险能力
- **影响**：后续若进入实际编码，应优先完成 P0 模块，不要一开始并行铺太多增强项

### D-008：Bitable 建表采用“两轮建设”策略
- **结论**：先建 7 张表的骨架字段和测试数据，再补关联、查找引用、公式字段
- **原因**：先保证主流程可录入、可联调，能显著降低飞书字段依赖带来的返工成本
- **影响**：建表实施阶段不追求一步到位，优先保证可跑与可验收

### D-009：测试环境先独立创建 Bitable App，再逐表落骨架
- **结论**：已创建测试版 Bitable App `OpenClaw-多赛事管理系统-测试`，并先落地 `赛事配置表`
- **原因**：先把真实环境入口搭起来，后续每建一张表都能直接做最小验收，不再停留在纸面设计
- **影响**：后续其余 6 张表可继续在该 App 内按清单顺序推进

### D-010：战队主档案库在选手表之前先落地
- **结论**：已先创建 `战队主档案库`，并完成 2 支测试战队的最小验收
- **原因**：选手表、场次表、卡组表后续都会依赖战队主表，先落地可以减少关联返工
- **影响**：下一步可以顺势创建 `选手主档案库`，再逐步补关联字段

### D-011：选手表首轮允许“无定妆照测试数据 + 待补资料状态”
- **结论**：`选手主档案库` 已落地，当前测试记录在未上传定妆照时统一标记为 `待补资料`
- **原因**：首轮目的是先验证表结构、角色字段、状态字段和异常标记可用，不必卡在附件完整性上
- **影响**：后续进入真实联调前，再补上传附件并收紧资料完整度校验

### D-012：赛事对阵表首轮先用文本型“下一场MatchID”串联顺延链
- **结论**：`赛事对阵与赛果表` 已落地，当前使用文本字段 `下一场MatchID` 先串起 3 场测试赛程链
- **原因**：首轮先验证顺延链条、场次状态、比分字段和签到字段可用，比过早做自关联更稳
- **影响**：后续若确有必要，再把 `下一场MatchID` 升级为关联字段；当前实现已足够支撑首版顺延逻辑开发

### D-013：卡组表首轮先验证版本与审核流，不把附件完整性当阻塞条件
- **结论**：`卡组提交与审核表` 已落地，当前测试记录允许先不上传截图，以 `待审核` / `待校验` / `有效版本` 等状态字段先跑通结构
- **原因**：首轮优先验证版本号、审核状态、有效版本、公示状态这几个真正驱动流程的字段
- **影响**：后续进入真实提交流程时，再补截图上传与 Deckstring 实际校验联动

### D-014：管理员白名单先用当前用户做首个测试样本
- **结论**：`管理员白名单` 已落地，并使用当前用户 open_id 作为第一条启用中的测试管理员记录
- **原因**：这样能最快打通管理员指令链路的身份校验前置条件，避免后面测试命令时还要回头补表
- **影响**：下一步做管理员指令联调时，可以直接用当前用户身份做白名单验证

### D-015：7 张核心表第一轮骨架已全部落地，下一步进入“第二轮字段补强 or 代码骨架”阶段
- **结论**：测试版 Bitable App 中 7 张核心表已全部创建，并各自完成了最小测试数据验收
- **原因**：项目已经从纯文档设计阶段正式进入真实数据结构阶段，具备继续补关联字段或直接接代码的条件
- **影响**：后续推进不再需要围绕“表有没有建”反复讨论，可以直接进入第二轮建模或代码实现

### D-016：先起最小代码骨架，再逐步把真实逻辑填进去
- **结论**：已创建 `app/` 代码骨架，先落本地启动入口、赛事上下文配置、指令路由骨架和 Bitable client 占位层
- **原因**：先把目录结构和主入口立起来，后续实现管理员指令、白名单校验、数据访问时会顺很多，也更不容易散
- **影响**：下一步将优先把占位层替换成真实的管理员白名单校验和基础指令实现

### D-017：先用本地 mock 数据跑通首批管理员指令，再替换成真实 Bitable 读写
- **结论**：当前已在本地 mock 数据层上跑通管理员白名单校验、`/检查注册`、`/暂停顺延`、`/恢复顺延`
- **原因**：先验证命令分层、路由结构、审计写入和返回文案是通的，再接真实 Bitable 会更稳、更容易定位问题
- **影响**：下一步的重点不再是设计命令接口，而是把 mock client 替换成真实数据读写实现

### D-018：真实 Bitable 访问先做成可切换模式，不在缺凭据时硬连
- **结论**：已增加 `BITABLE_CLIENT_MODE=mock|real` 双模式；当前因宿主环境缺少 `FEISHU_APP_ID / FEISHU_APP_SECRET`，real 模式只完成代码接线，未做实际验证
- **原因**：先把真实访问的接入点和报错边界做好，比在当前环境里硬拼一个半通不通的 real 模式更稳
- **影响**：后续只要补齐飞书应用凭据，就可以优先验证 `/暂停顺延`、`/恢复顺延` 的 real 模式链路
