# 🏆 OpenClaw x 飞书机器人：多赛事全链路管理系统需求文档（PRD v1.1）

## 0. 文档说明
本版本基于初版 PRD 补强，目标是让方案更适合直接进入数据建模、机器人逻辑设计与开发排期阶段。

**本轮确认范围：**
- 保留“多赛事隔离、队长全责制、比分驱动顺延、人工风控开关”四大核心逻辑
- **权限模型不扩展**，先按“管理员白名单 + 队长点击限制”执行
- 优先补强：状态机、数据结构、顺延触发逻辑、异常处理、审计留痕

---

## 1. 系统核心逻辑
- **多赛事隔离**：系统需支持 2-3 个赛事群并发运行，确保数据、通知、指令、卡片回调互不串号。
- **队长全责制**：机器人所有通知、催缴、签到提醒默认只触达队长，普通队员不主动打扰。
- **比分驱动顺延**：通过监听上一场赛果，在满足触发条件时自动下发下一场签到卡片。
- **人工风控开关**：档案检查、卡组公示、卡组审核、暂停/恢复顺延等关键动作均保留人工控制权。
- **全链路可追溯**：所有关键动作需记录操作人、来源、时间与结果，便于争议复盘。

---

## 2. 核心业务目标
1. 提高赛事运营效率，减少人工逐个催办。
2. 保证多赛事并行时不会串群、串队、串场次。
3. 让签到与顺延机制足够稳定，避免重复推送与误触发。
4. 保留人工裁判和运营对高风险动作的最终控制权。
5. 支持后续扩展为更多赛事模板和复用型工作流。

---

## 3. 关键对象与唯一标识
为避免后期数据与回调混乱，建议在所有核心表中统一增加业务主键。

### 3.1 必备标识
- **TournamentID**：赛事唯一 ID，作为业务主键。
- **ChatID**：飞书群 ID，用于识别消息来源。
- **MatchID**：场次唯一 ID。
- **TeamID**：战队唯一 ID。
- **PlayerID**：选手唯一 ID（可直接复用飞书 UserID 或内部生成）。
- **SubmissionID**：卡组提交批次唯一 ID。

### 3.2 多赛事隔离原则
- 运行态可使用 `Map<ChatID, TournamentID>` 做快速映射。
- 数据层必须保留 `TournamentID` 字段，不能只依赖 ChatID。
- 所有卡片回调、指令执行、事件消费都必须携带 TournamentID 进行校验。

## 6. 管理员指令集（白名单制）
机器人仅响应配置在 Bitable《管理员白名单》中的用户指令。

| 指令 | 作用描述 | 机器人反馈 |
| :--- | :--- | :--- |
| `/检查注册` | 全群扫描注册情况 | 汇总未注册/未完成/异常名单，在群内 **仅 @ 对应战队队长** 催促。 |
| `/公示卡组` | 解除保密机制 | 将卡组公示视图链接发送至赛事群。 |
| `/暂停顺延` | 应对突发重赛 | 挂起比分监听逻辑，不再自动开启下一场签到。 |
| `/恢复顺延` | 解除暂停状态 | 重新启用比分驱动顺延。 |
| `/手动签到 [场次]` | 强制干预 | 手动为特定场次下发签到卡片。 |
| `/重发战报 [场次]` | 失败兜底 | 重新发送指定场次战报。 |

**执行原则：**
- 机器人收到指令后，先验证发起人是否在白名单内。
- 再校验该指令是否属于当前赛事上下文。
- 所有指令动作需写入审计日志。

## 7. 自动化业务流

### 第一阶段：进场与注册
#### 触发方式
- 选手入群后，管理员发送 `/检查注册`。

#### 核心逻辑
- 机器人比对群成员 UserID 与选手主档案库。
- 识别以下异常：
  - 未建档
  - 已建档但资料缺失
  - 队伍人数不足
  - 队员已报名但未入群
  - 入群账号与登记账号不一致

#### 输出动作
- 机器人汇总异常，并 **仅 @ 对应战队队长** 发送催缴提醒。
- 提醒内容优先以“战队维度”汇总，不单独点名队员。

### 第二阶段：卡组收集与审核
#### 提交方式
- 战队队长提交全队卡组代码与卡组图片。

#### 核心逻辑
- 每次提交生成一条新的 Submission 记录，不覆盖旧记录。
- 新记录默认状态为“已提交待审核”。
- OpenClaw 负责对 Deckstring 做基础合法性校验。
- 运营在 Bitable 后台人工核对代码与截图后，将状态手动改为“审核通过”或“审核驳回”。

#### 输出动作
- 审核通过：可进入待公示状态。
- 审核驳回：记录驳回原因，并通知队长重新提交。

### 第三阶段：赛中顺延签到（Rolling Schedule）
#### 触发点 1：首场签到
- 由管理员手动开启或定时开启。

#### 触发点 2：顺延触发
- 机器人监听《赛事对阵与赛果表》。
- 当上一场进入赛点状态，且满足顺延触发条件时，下发下一场签到卡片。

#### 顺延触发条件
必须同时满足：
1. 当前赛事未被暂停；
2. 上一场 MatchStatus = `进行中`；
3. 上一场比分**首次**达到赛点阈值；
4. 下一场 MatchStatus = `未开启` 或 `待签到`；
5. 下一场尚未触发过顺延；
6. 本次事件未命中重复触发幂等检查。

#### 签到卡片规则
- 机器人推送交互式消息卡片，消息体 **仅 @ 双方战队队长**。
- 卡片按钮 payload 必须包含：TournamentID、MatchID、Action、Version。
- 仅对应队长点击有效。
- 非队长点击时，返回“你不是本场签到负责人”。
- 已签到后重复点击，返回幂等提示，不重复写数据。

#### 签到完成判定
建议规则：
- A 队长点击后仅标记 A 队已签到；
- B 队长点击后仅标记 B 队已签到；
- **双方都已签到** 时，场次进入 `已签到待开赛`。

#### 超时处理
- 若卡片发出后 X 分钟未完成双边签到，机器人再次 **仅 @ 双方队长** 提醒。
- 若多次超时未处理，转人工介入。

### 第四阶段：结果录入与战报公示
#### 录入方式
- 裁判在 Bitable 中录入比分。

#### 核心逻辑
- 机器人监听比分字段变更。
- 更新场次状态与战报发送状态。
- 若该场满足自动公示条件，则发送战报海报或文本战报。

#### 输出动作
- 优先发送战报海报（包含大局/小局比分）。
- 若海报生成失败，则降级发送文本战报。
- 若发送失败，写入失败状态并支持 `/重发战报 [场次]`。

## 8. 顺延逻辑的稳定性要求
比分驱动顺延是整个系统最关键、也最容易出事故的部分，必须明确以下机制：

### 8.1 幂等机制
建议生成唯一触发键：
`trigger_key = TournamentID + current_match_id + next_match_id + trigger_type`

若同一触发键已处理成功，则后续重复事件直接忽略。

### 8.2 防抖机制
- 对比分更新事件设置 3~5 秒防抖窗口。
- 同一场次短时间多次更新时，只处理最后一次稳定状态。

### 8.3 乱序容错
- 若收到事件时发现当前状态不满足前置条件，则不执行顺延。
- 必要时可回查当前 Bitable 记录的最终状态，而不是完全信任单次 webhook 载荷。

### 8.4 暂停恢复
- `/暂停顺延` 后，所有自动顺延动作停止执行。
- `/恢复顺延` 后，仅对后续事件继续生效，不自动补发历史漏掉场次。
- 历史漏发场次由 `/手动签到 [场次]` 兜底。

## 9. 技术部署要求
- **OpenClaw 集成**：使用 OpenClaw 校验 Deckstring 与编排机器人逻辑。
- **长连接订阅**：至少订阅 `im.message.receive_v1` 与 `bitable.table_record_updated_v1` 事件。
- **多赛事识别**：代码中需维护 `Map<ChatID, TournamentID>` 或等价配置映射。
- **卡片回调校验**：所有卡片点击都必须校验用户、赛事、场次、版本。
- **事件幂等处理**：所有记录变更、顺延触发、战报发送都必须可重复消费而不重复执行。
- **可恢复性**：服务重启后能够通过表中状态恢复未完成任务，而不是仅依赖内存态。
- **环境隔离**：测试赛事群 / 测试 Bitable / 测试卡片模板需与生产环境隔离。

---

## 10. 消息与卡片设计原则
- 所有通知尽量按“战队维度”表达，而不是点对点骚扰个体队员。
- 卡片文案要明确反馈当前状态，例如：
  - 已签到
  - 对方尚未签到
  - 本场已开赛
  - 本赛事顺延已暂停
- 所有失败反馈必须可读，不仅返回“失败”。
- 卡片模板建议支持版本号，避免老卡片 payload 与新逻辑不兼容。

## 11. 异常与兜底策略
### 11.1 注册异常
- 队员未入群
- 队员入群但未绑定
- 战队人数不完整
- 队长缺失
- 账号与报名信息不一致

### 11.2 卡组异常
- Deckstring 非法
- 职业数量不足
- 图片缺失
- 图片与代码不一致
- 审核驳回后未重新提交

### 11.3 赛程异常
- 比分回滚
- 场次取消
- 重赛
- 手动改场序
- 顺延暂停后恢复

### 11.4 消息异常
- 卡片发送失败
- 卡片重复点击
- 非队长点击
- 战报发送失败
- webhook 重复推送

---

## 12. 验收重点（开发自测优先项）
1. **只有对应队长能点签到**。
2. **双边签到后场次状态正确更新**。
3. **比分首次达到赛点时才自动触发下一场卡片**。
4. **同一比分事件重复推送不会重复发卡**。
5. **暂停顺延后自动顺延完全停止**。
6. **恢复顺延后新事件可继续工作，历史漏发需手动补发**。
7. **战报生成失败时可自动降级发文本**。
8. **多赛事并行时不同 ChatID / TournamentID 不串数据**。

## 14. 本版关键结论
相比初版 PRD，本版重点补上了：
- TournamentID 作为业务主键
- 更细的状态机
- Match 表的结构化比分字段
- Deck Submission 的批次化设计
- 审计日志表
- 顺延触发的幂等、防抖、暂停恢复机制
- 战报失败兜底与异常清单

以上补强可以显著降低后期返工风险，并让开发更容易直接开始实现。
