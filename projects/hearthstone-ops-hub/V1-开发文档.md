# Hearthstone Ops Hub V1 开发文档

## 1. 项目定位

Hearthstone Ops Hub 是一个面向 **炉石社区运营场景** 的轻量工作台。  
它不是通用项目管理工具，而是专门解决下面这类问题：

- 赛事、活动排期很多，时间交叉
- 很多任务要围绕关键日期提前准备
- 每天都有明确待办，需要快速看到“今天该做什么”
- 运营文档信息很多，但执行时需要被拆成可勾选、可追踪的任务

V1 的核心目标只有一句话：

**把《2025年4月工作规划》里的赛事、活动、每日待办，变成一个可以每天直接使用的运营执行面板。**

---

## 2. V1 产品目标

V1 不追求“大而全”，只先完成最有价值的 4 件事：

1. **录入并管理赛事 / 活动**
2. **录入并查看每日待办**
3. **用日历 + 今日面板查看排期**
4. **支持任务状态更新并上线可用**

也就是说，V1 先回答两个最现实的问题：

- **今天我要做什么？**
- **这个月有哪些关键比赛 / 活动 / 上线节点？**

---

## 3. V1 范围

### 3.1 包含内容

#### A. 赛事 / 活动管理
支持录入以下类型的运营对象：
- 赛事
- 活动
- 节点型事项（如某天上线、提需、配置、复盘）

每条数据至少包含：
- 名称
- 类型
- 开始日期
- 结束日期
- 优先级
- 状态
- 备注

---

#### B. 任务管理
支持维护任务，并与赛事 / 活动关联。

每条任务至少包含：
- 任务标题
- 所属赛事 / 活动
- 日期
- 优先级
- 状态
- 任务类型
- 备注

状态先做最基础的 4 个：
- 待处理
- 进行中
- 已完成
- 已逾期

---

#### C. 每日执行面板
首页做成运营工作台，而不是普通列表页。

首页至少展示：
- 今日待办
- 逾期任务
- 未来 3 天高优先级任务
- 今日相关活动 / 赛事
- 本周关键节点

---

#### D. 日历视图
提供月历视图，集中展示：
- 赛事日期
- 活动日期
- 每日待办数量
- 关键任务节点

作用不是编辑，而是 **看排期冲突和时间分布**。

---

#### E. 数据初始化
将《网易大神炉石传说社区运营 - 2025年4月工作规划》里的内容整理后导入系统，包括：
- 6 场赛事
- 多个活动
- 4 月 1 日至 30 日的每日待办

---

#### F. 部署上线
V1 完成后部署到 Vercel，可直接访问使用。

---

### 3.2 V1 暂不包含

以下内容明确放到后续版本，不进入 V1：

- 复杂模板引擎
- 自动根据 T 日批量生成任务
- 多人协作与细粒度权限
- 评论 / 操作日志系统
- 通知提醒（邮件 / 飞书）
- 域名购买与正式品牌包装
- 数据统计报表
- 移动端复杂适配

原因很简单：  
这些都重要，但**不是第一版能不能真正用起来的关键**。

## 4. 用户场景

### 场景 1：每天打开看今天要做什么
用户进入首页，直接看到：
- 今天必须处理的任务
- 有没有逾期事项
- 今天涉及哪些赛事 / 活动
- 接下来 3 天有没有高优先级风险点

---

### 场景 2：查看本月整体排期
用户进入日历页，看到：
- 4.10 - 4.12 世冠赛春季平台赛决赛
- 4.18 - 4.19 黄金棋坛
- 4.3 冲钻领卡包活动上线
- 4.1 世冠赛平台赛决赛美术提需

这样可以快速理解：
**哪几天最忙，哪些事项容易撞车。**

---

### 场景 3：更新执行状态
当某项工作完成后，用户在任务列表或今日面板中直接勾选完成。  
系统即时更新状态，不需要跳很多页面。

---

### 场景 4：查看某个活动的完整上下文
点击某场赛事 / 某个活动后，可以看到：
- 基本信息
- 时间范围
- 关联任务
- 当前进展
- 备注说明

---

## 5. 产品结构

V1 建议只做 4 个核心页面。

### 5.1 Dashboard / 首页
模块：
- 今日待办
- 已逾期
- 未来三天高优先级
- 今日相关赛事 / 活动
- 本周关键节点

这是最重要的页面，优先级最高。

---

### 5.2 Calendar / 日历页
功能：
- 月历展示
- 按天显示赛事 / 活动 / 任务数量
- 点击日期查看当天任务详情

---

### 5.3 Tasks / 任务页
功能：
- 查看全部任务
- 按状态筛选
- 按优先级筛选
- 按赛事 / 活动筛选
- 修改状态

---

### 5.4 Events / 活动页
功能：
- 查看全部赛事 / 活动
- 查看详情
- 查看关联任务
- 编辑基础信息

---

## 6. 数据模型设计（V1）

V1 不做过度抽象，先采用最稳的 3 张核心表。

---

### 6.1 events
用于存储赛事、活动、关键运营对象。

字段建议：

- `id`
- `title`：名称
- `type`：event / campaign / milestone
- `start_date`
- `end_date`
- `priority`：critical / high / medium / low
- `status`：planned / active / done / cancelled
- `description`
- `created_at`
- `updated_at`
- `user_id`

说明：
- `event`：赛事
- `campaign`：活动
- `milestone`：关键节点型事项

---

### 6.2 tasks
用于存储执行任务。

字段建议：

- `id`
- `title`
- `event_id`：关联 events
- `task_date`：任务对应日期
- `planned_date`：计划开始处理日期，可为空
- `due_date`：截止日期，可为空
- `task_type`：design / config / launch / content / review / data / misc
- `priority`：critical / high / medium / low
- `status`：pending / in_progress / done / overdue
- `source_type`：manual / imported
- `notes`
- `created_at`
- `updated_at`
- `user_id`

说明：
- `task_date` 用来挂到某一天的执行面板上
- `planned_date` 和 `due_date` 可以先保留，方便后续升级自动化

---

### 6.3 daily_checklists
用于承接“4 月 1 日到 30 日的每日待办原文”。

字段建议：

- `id`
- `check_date`
- `title`
- `content`
- `priority`
- `status`
- `related_event_id`
- `created_at`
- `updated_at`
- `user_id`

说明：
这张表不是必须永远存在，但 **V1 很适合保留**。  
因为你现在最大的真实输入源，就是“每日待办清单”，不必强行把它全部塞进 tasks。

## 8. 开发原则

### 原则 1：先能用，再优雅
先把文档里的内容变成结构化数据并展示出来，  
不要一开始就陷入模板系统、自动化规则、复杂抽象。

### 原则 2：首页优先
首页必须成为真正的“运营驾驶舱”。  
不是欢迎页，也不是数据大屏。

### 原则 3：录入真实业务数据优先于搭框架
优先把 4 月规划内容录进去，系统立刻有价值。  
不要只把项目骨架搭好，却没有真实数据可用。

### 原则 4：为 V2 留扩展点，但不提前实现
例如：
- `planned_date`
- `due_date`
- `source_type`
- `task_type`

这些字段可以为后续自动化做准备，但 V1 不需要把完整模板引擎做出来。
