# 赛事主控 Agent First 30 Days Plan v1

> 用途：定义赛事主控 agent 上线后的首轮工作重点，避免一开始就无限扩张范围。

---

## Day 1-3：理解与校验

目标：先看懂，不着急大改。

- [ ] 读完 core / rules / checklist / bootstrap / SOP index
- [ ] 理解文档结构与关键代码入口
- [ ] 核对飞书表结构与关键字段
- [ ] 确认当前最成熟链路是个人赛赛果链路
- [ ] 输出一版“当前系统认知摘要”给管理员确认

---

## Day 4-7：打通最小闭环

目标：在测试环境中完成一次最小真实闭环。

- [ ] 跑通测试赛事入口
- [ ] 验证回调地址、token、url_verification
- [ ] 验证至少一轮真实卡片回调
- [ ] 验证至少一轮真实写表
- [ ] 验证日志可追踪

---

## Week 2：灰度承接

目标：小范围开始接真正的赛务动作。

- [ ] 只接一个测试赛事或低风险赛事
- [ ] 承接标准化提醒、报名、卡组或赛果中的部分节点
- [ ] 输出每日简短待办/异常摘要
- [ ] 建立“哪些动作必须升级人工”的稳定边界

---

## Week 3：稳定运行

目标：把常见动作跑稳。

- [ ] 常见提醒类动作稳定运行
- [ ] 常见审核推进稳定运行
- [ ] 赛果/争议相关动作不再依赖个人助理上下文
- [ ] 出现异常时能给出结构化升级报告

---

## Week 4：评估扩展

目标：决定是否进入正式接管更多赛事。

- [ ] 复盘灰度期间的异常与回滚情况
- [ ] 确认是否出现串线/错写/职责漂移
- [ ] 确认是否具备承接更多赛事的稳定性
- [ ] 输出“是否进入正式扩容”的建议

---

## 30 天内不要做的事

- [ ] 不要一上来重构整个系统
- [ ] 不要直接替换所有现有入口
- [ ] 不要在未灰度时同时承接多个正式赛事
- [ ] 不要把私人助理上下文搬进赛事实例
- [ ] 不要为了体验优化先牺牲稳定性

---

## 一句话目标

> 前 30 天的目标不是做大，而是证明：这个赛事主控 agent 能稳定、清晰、低串线地独立运行。
