# 赛事主控 Agent Bootstrap Context v1

> 用途：新赛事主控代理首次启动时优先阅读，用最短路径理解当前炉石赛事系统现状。

---

## 1. 你将接管什么

你将接管的是：
- 炉石赛事自动化系统的赛事运营与执行主控
- 赛事流程推进
- 群入口与卡片回调协同
- 结构化数据驱动的赛务闭环

你不会接管：
- 用户私人助理职责
- 用户长期私人记忆
- 与赛事系统无关的工作

---

## 2. 当前系统大致状态

当前系统不是从零开始，而是已经完成了一部分关键建设。

### 已完成的关键方向
- 飞书多维表格作为主要结构化数据层
- 赛事流程核心代码已有基础实现
- 个人赛赛果链路关键分支已完成真实闭环样本验证
- 飞书卡片回调最小 HTTP 入口已补齐
- 飞书事件适配层已补齐
- 本地 dry-run / real 联调路径已形成

### 当前最值得信任的事实
- 当前系统已经不只是纸面设计，而是已有真实写表验证样本
- 个人赛赛果链路是目前最成熟的真实闭环部分之一
- 线上接入已开始推进，但尚未完成“新赛事主控 + 新 bot”正式接管

---

## 3. 当前最重要的业务结论

### 个人赛赛果链路
已形成三条关键真实闭环样本：
1. `submit -> confirm`
2. `submit -> reject`
3. `submit -> timeout -> admin-confirm`

这意味着：
- 对手确认分支可用
- 拒绝升级 dispute 分支可用
- timeout 后管理员兜底收口可用

### 特别注意
如果 `admin-confirm` 需要同步收口 dispute，必须：
- 使用 timeout/reject 之后的最新 report 快照
- 带上 `linked_dispute_uid`

否则可能只完成部分写回，导致 disputes 未完整更新。

## 6. 当前系统尚未完全完成的部分

你必须知道：
- 多赛事独立 session/agent 的运行时隔离还未正式落地
- 新赛事专用飞书 bot 还未完成切流
- 总后台/跨赛事主控台还不是完整产品形态
- 请求签名校验、幂等去重、卡片状态回写还可继续加强

因此，你的第一阶段目标不是“重构一切”，而是：
- 在不破坏已有可用链路的前提下接管
- 先稳住报名/卡组/赛果/争议四大主链路
- 再逐步扩展体验与多赛事并行能力

---

## 7. 当前推荐接管顺序

### 第 1 阶段：理解与验证
- 理解文档体系
- 核实表结构与字段
- 跑通测试赛事最小链路
- 验证新 bot / 新入口 / 新实例边界

### 第 2 阶段：灰度接管
- 只接测试赛事
- 只接一个测试群
- 只接一位管理员入口
- 确认所有关键写表动作可追溯

### 第 3 阶段：正式接管
- 新赛事逐步进入新主控体系
- 旧体系保留观察期与回滚路径

---

## 8. 你的第一反应应是什么

不是立刻改代码，也不是立刻改配置。
而是先回答这几个问题：
- 当前处理的是哪个赛事？
- 当前链路处于哪个阶段？
- 当前事实来自哪张表/哪份文档？
- 当前动作是否会影响现网？
- 当前动作是否可回滚？

---

## 9. 一句话启动提醒

> 你接管的是一个已经有真实闭环样本的赛事系统；先理解、先验证、先灰度，不要为了追求完整而一次性大改现网。
