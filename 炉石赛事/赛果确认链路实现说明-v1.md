# 赛果确认链路实现说明 v1

> 日期：2026-03-28

## 1. 本轮目标

把个人赛整场结果从“只在 `matches` 里有状态”再往前推进一层，补出：

- 一方上报整场结果
- 对手确认 / 拒绝
- 管理员兜底确认

采用最小可用实现，先把状态机和写回链路接好。

---

## 2. 已新增 action

已在 `callback-handler.js` 中新增：

- `submit_match_result_report`
- `confirm_match_result_report`
- `reject_match_result_report`
- `admin_confirm_match_result_report`

---

## 3. 当前行为

### 3.1 submit_match_result_report
- 参赛选手可提交整场结果
- 自动识别 reporter_side（side_a / side_b）
- 若 `matches.final_result_text` 已存在，则优先复用
- 否则尝试根据 `conquest_state_json` 生成结果文案
- 写入 `result_reports`（若已配置表）
- 同时把 `matches` 改为：
  - `result_status = pending_opponent_confirmation`
  - `match_status = result_pending_confirmation`

### 3.2 confirm_match_result_report
- 仅对手方可确认
- 确认后：
  - `result_reports.report_status = opponent_confirmed`
  - `matches.result_status = confirmed`
  - `matches.match_status = completed`
- 会尝试群内发送确认结果

### 3.3 reject_match_result_report
- 仅对手方可拒绝
- 拒绝后：
  - `result_reports.report_status = opponent_rejected`
  - `matches.result_status = disputed`
  - `matches.match_status = disputed`

### 3.4 admin_confirm_match_result_report
- 管理员 / 裁判兜底确认
- 用于：
  - 对手超时未确认
  - 赛事走 judge_only 模式
  - 临时人工接管
- 确认后：
  - `result_reports.report_status = admin_confirmed`
  - `matches.result_status = confirmed`
  - `matches.match_status = completed`

---

## 4. 当前依赖

### 4.1 可选表
通过环境变量读取：

- `HS_RESULT_REPORTS_TABLE_ID`

若未配置：
- `submit_match_result_report` 仍可先推动 `matches` 状态
- 但确认 / 拒绝 / 管理员兜底这几步将缺少真实 report 载体
- 因此下一步仍建议补建 `result_reports` 表

### 4.2 选手身份识别
当前实现默认从 `matches` 上读取：

- `side_a_entity_uid` / `side_a_uid` / `player_a_uid` / `side_a_player_uid`
- `side_b_entity_uid` / `side_b_uid` / `player_b_uid` / `side_b_player_uid`

再去 `players` 中找对应的 `feishu_open_id`。
