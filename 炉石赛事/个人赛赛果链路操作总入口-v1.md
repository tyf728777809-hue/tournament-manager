# 个人赛赛果链路操作总入口 v1

> 日期：2026-03-29

## 1. 这份文档是干什么的

这是个人赛赛果链路的**总入口**。

目标：以后无论是做真实联调、补 write_plan、还是排查某个分支，都优先从这份文档进入，而不是到处翻散落说明。

它负责把以下内容串起来：

- prefetched 组装器
- prefetched 统一入口
- confirm 模板
- reject 模板
- timeout 模板
- admin-confirm 模板

---

## 2. 当前推荐工作流（总览）

统一按这条线走：

1. **当前会话读真实表**
2. **组装 `prefetched.json`**
3. **统一入口生成 `write_plan`**
4. **当前会话串行执行 `write_plan`**
5. **重新读表复核结果**

也就是：

```text
真实表查询结果
  -> build-solo-result-prefetched.js
  -> prefetched.json
  -> run-callback-with-prefetched.js
  -> plan.json
  -> feishu_bitable_app_table_record 串行执行
  -> 落表复核
```

## 4. 脚本索引

### 4.1 prefetched 组装器

```bash
node scripts/build-solo-result-prefetched.js
```

作用：
- 输入原始查询结果文件
- 输出标准 `prefetched.json`
- 自动推导 `sideContext`

### 4.2 prefetched 统一入口

```bash
node scripts/run-callback-with-prefetched.js
```

作用：
- 输入 `callback.json + prefetched.json (+ context.json)`
- 输出 `result.json + plan.json`
- 可选 `--render-plan` 输出渲染后的工具调用规格

### 4.3 write_plan 渲染器

```bash
node scripts/render-feishu-write-plan.js
```

作用：
- 把已有 `plan.json` 渲染成更直观的工具调用清单
- 适合执行前复核

## 8. 推荐执行顺序

如果是完整演练，建议按这个顺序：

1. `submit`
2. `confirm`
3. `reject`
4. `timeout`
5. `admin-confirm`

但真实联调时通常不会一条对局把所有状态都跑一遍。

更实际的用法是：
- 用一条样本专门跑 `confirm`
- 用一条样本专门跑 `reject -> admin-confirm`
- 或者用复位后的样本分别跑 `timeout -> admin-confirm`

---

## 9. 成功标准

无论哪条分支，统一按这 4 条判断是否跑通：

1. `result.json` 成功返回
2. `plan.json` 条数和顺序符合预期
3. 当前会话串行执行 `write_plan` 全部成功
4. 重新读表后，目标表状态与模板预期一致
