# 赛事主控 Agent Handover Checklist v1

> 用途：在新赛事主控代理正式接管前，逐项确认已掌握的信息、边界与现状。

---

## 1. 身份与边界确认

- [ ] 明确自己是“赛事主控代理”，不是用户私人助理
- [ ] 明确自己不继承私人助理长期记忆作为业务真相
- [ ] 明确只处理赛事相关任务
- [ ] 明确个人助理与赛事主控的分工边界
- [ ] 明确需要使用独立飞书机器人承接赛事入口

---

## 2. 数据真相源确认

- [ ] 已知赛事系统的结构化真相源是飞书多维表格 + 飞书文档
- [ ] 已知核心表范围（tournaments / registrations / players / teams / deck_submissions / decks / matches / result_reports / disputes）
- [ ] 已知文档体系位置和用途
- [ ] 已知聊天上下文不能替代结构化真相

---

## 3. 当前系统已完成能力

### 3.1 已完成 / 已验证
- [ ] 已知个人赛赛果链路已完成真实闭环样本验证
- [ ] 已知 confirm / reject / timeout->admin-confirm 三条关键分支已收口
- [ ] 已知本地最小 HTTP webhook 入口已存在
- [ ] 已知飞书事件适配层已存在
- [ ] 已知 dry-run 验证模式可用

### 3.2 当前可复用资产
- [ ] 已知 `callback-handler.js` 是当前核心业务入口
- [ ] 已知 `bitable-api-utils.js` 负责表操作
- [ ] 已知 `notification-utils.js` 负责通知能力
- [ ] 已知已有模板、摘要、联调清单、操作总入口等文档可复用

---

## 4. 当前系统未完全完成事项

- [ ] 不是所有赛事链路都已完全产品化
- [ ] 多赛事独立 agent/session 运行时隔离尚未正式落地
- [ ] 总后台 / 跨赛事主控台尚未完整成型
- [ ] 新赛事主控实例与新飞书机器人尚未正式接管现网
- [ ] 请求签名校验 / 幂等层 / 卡片回写体验等仍可继续增强

---

## 5. 当前最重要的业务注意点

- [ ] 若 admin-confirm 要同步收口 dispute，必须使用 timeout/reject 之后的最新 report 快照
- [ ] 若缺少 `linked_dispute_uid`，可能无法完整更新 disputes
- [ ] 处理赛果时必须优先确保状态一致性，而不是只追求卡片动作跑通
- [ ] 任何关键写表动作前都要再次确认赛事归属与记录版本

---

---

## 7. 正式接管前的配置确认

- [ ] 将使用独立 OpenClaw 实例或独立运行边界
- [ ] 将使用独立飞书机器人，而不是沿用个人助理 bot
- [ ] 已确认赛事主控的 workspace / config / memory 与个人助理隔离
- [ ] 已确认新 bot 的权限范围与接入群范围
- [ ] 已确认回滚路径（如新主控异常，如何临时兜底）

---

## 8. 灰度接管建议

正式接管前，建议先完成：
- [ ] 一个测试赛事
- [ ] 一个测试群
- [ ] 一个管理员私聊
- [ ] 一轮真实卡片回调
- [ ] 一轮真实写表校验
- [ ] 一轮异常兜底演练

全部通过后，再进入正式接管。

---

## 9. 正式接管判定标准

只有当以下条件全部满足，才算“赛事主控可正式接管”：

- [ ] 职责边界明确
- [ ] 数据边界明确
- [ ] 新 bot 可独立收消息并回调
- [ ] 关键赛事链路可独立推进
- [ ] 不依赖私人助理上下文即可运行
- [ ] 出现异常时有升级与回滚方案

---

## 10. 接管后第一阶段工作重点

- [ ] 先接一个测试赛事，不要一上来全量接管
- [ ] 先稳定报名 / 卡组 / 赛果 / 争议四大主链路
- [ ] 先把日志、摘要、待办、异常升级机制跑稳
- [ ] 再考虑多赛事并行与体验优化

---

## 11. 一句话交接目标

> 接管的不是用户私人上下文，而是“炉石赛事系统的可运行操作体系”。
