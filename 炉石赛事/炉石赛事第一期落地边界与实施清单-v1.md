# 炉石赛事第一期落地边界与实施清单 v1

更新时间：2026-03-29

---

## 1. 第一期目标

### 1.1 核心目标

把“可稳定上线”的主链路跑通，确保：

1. 报名能稳定收集
2. 卡组能稳定提交
3. 赛果能正常记录
4. OpenClaw 能查、能提醒、能解释

### 1.2 第一期不追求

- 完美卡片体验闭环
- 所有交互都在卡片内完成
- 高度自动化的裁判系统

### 1.3 第一期优先级

**稳定 > 体验 > 自动化**

---

## 2. 能力分层决策

### 2.1 报名入口

**决策：迁移到飞书表单/H5**

| 维度 | 结论 |
|------|------|
| 主承载 | 飞书表单 或 独立 H5 页面 |
| 入口 | 群卡片按钮打开表单链接 |
| 回调 | 不依赖 OpenClaw 卡片回调 |
| 写表 | 表单提交后由业务服务写 Bitable |
| OpenClaw 职责 | 查询报名状态、审核辅助、提醒 |

**原因**：
- 字段多、需要修改、有资料复用
- 表单天然适合结构化采集
- 不受卡片回调稳定性影响

---

### 2.2 卡组提交入口

**决策：迁移到飞书表单/H5**

| 维度 | 结论 |
|------|------|
| 主承载 | 飞书表单 或 独立 H5 页面 |
| 入口 | 报名审核通过后由业务服务发入口卡片 |
| 回调 | 不依赖 OpenClaw 卡片回调 |
| 写表 | 表单提交后由业务服务写 Bitable |
| OpenClaw 职责 | 查询提交状态、审核说明、异常解释 |

**原因**：
- 有图片、代码、版本管理
- 需要校验职业、去重
- 交互复杂，不适合塞在卡片里

---

### 2.3 赛果上报与确认

**决策：分两阶段处理**

#### 第一期方案

| 维度 | 结论 |
|------|------|
| 上报方式 | 选手私聊机器人上报 / 裁判代填 |
| 确认方式 | 管理员后台确认 或 群消息确认 |
| 主承载 | OpenClaw 对话 + Bitable 视图 |
| 卡片按钮 | 第一期不依赖卡片确认闭环 |

#### 第二期目标（不在第一期范围）

- 业务服务发送确认卡片
- 业务服务自己接回调
- 选手点击确认按钮完成闭环

**原因**：
- 这是卡片按钮价值最高的场景
- 但第一期优先保证链路稳定
- 第二期再把卡片体验补齐

---

### 2.4 争议提交

**决策：迁移到飞书表单/H5**

| 维度 | 结论 |
|------|------|
| 主承载 | 飞书表单 或 独立 H5 页面 |
| 入口 | 私聊机器人引导 / 群卡片链接 |
| 回调 | 不依赖 OpenClaw 卡片回调 |
| 写表 | 表单提交后由业务服务写 Bitable |
| OpenClaw 职责 | 查询争议状态、AI 辅助分析、裁决建议 |

**原因**：
- 需要长文本、附件、证据
- 表单最适合

---

### 2.5 管理员审核与运营

**决策：保留在 OpenClaw + Bitable 视图**

| 维度 | 结论 |
|------|------|
| 列表视图 | Bitable 视图 |
| 审核动作 | OpenClaw 对话 + 工具调用 |
| 统计查询 | OpenClaw 查询并解释 |
| 提醒 | OpenClaw 主动提醒 |

**OpenClaw 典型职责**：
- "帮我列出待审报名"
- "帮我看这场争议怎么判"
- "谁还没交卡组"
- "帮我发群公告"
- "帮我提醒 XXX 补材料"

## 3. 技术实施边界

### 3.1 业务服务 vs OpenClaw 边界

#### 业务服务（第一期新建或复用现有）

负责：
- 报名表单提交后写 `players / registrations / registration_members`
- 卡组表单提交后写 `deck_submissions / decks`
- 争议表单提交后写 `disputes`
- 后续：赛果确认卡片回调处理

实现方式（二选一）：
- **方案 A**：复用当前 `feishu-callback-server.js`，增强表单回调处理
- **方案 B**：新建独立 Node 服务，专门处理表单回调

#### OpenClaw

负责：
- 查询 Bitable 并解释
- 发送提醒
- 审核辅助建议
- 争议 AI 分析
- 群公告草拟
- 运营对话

---

### 3.2 数据落地边界

所有结构化数据统一写 Bitable，遵循：

| 表 | 写入方 | 读取方 |
|---|--------|--------|
| players | 业务服务 | OpenClaw + 业务服务 |
| teams | 业务服务 | OpenClaw + 业务服务 |
| registrations | 业务服务 | OpenClaw + 业务服务 |
| deck_submissions | 业务服务 | OpenClaw + 业务服务 |
| decks | 业务服务 | OpenClaw + 业务服务 |
| matches | OpenClaw（第一期） | OpenClaw + 业务服务 |
| result_reports | OpenClaw（第一期） | OpenClaw + 业务服务 |
| disputes | 业务服务 | OpenClaw + 业务服务 |

## 6. 风险与应对

### 6.1 表单/H5 部署风险

**风险**：需要额外部署和维护

**应对**：
- 第一期可先用飞书原生表单
- 后续再升级 H5

### 6.2 OpenClaw 查询能力依赖

**风险**：OpenClaw 需要稳定的 Bitable 查询能力

**应对**：
- 复用已有 `bitable-api-utils.js`
- 增加常用查询模板

### 6.3 多服务协调

**风险**：OpenClaw + 业务服务需要协调

**应对**：
- 第一期先做简单回调
- 明确数据写入边界
- 通过 Bitable 作为协调枢纽
