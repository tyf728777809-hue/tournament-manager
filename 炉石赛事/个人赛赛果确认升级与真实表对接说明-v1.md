# 个人赛赛果确认升级与真实表对接说明 v1

> 日期：2026-03-28

## 1. 本轮新增能力

本轮在已有 `submit_match_result_report / confirm / reject / admin_confirm` 基础上，补了 3 个关键点：

1. **对手确认从纯文本提示升级为正式卡片按钮**
   - 私聊对手时直接发送确认卡片
   - 卡片内含：
     - 赛事名
     - 对局双方
     - 对手提交的赛果
     - 确认截止时间
     - `确认赛果` / `拒绝并发起争议` 按钮

2. **补超时升级管理员处理**
   - 新增 action：`escalate_match_result_report_timeout`
   - `submit_match_result_report` 时自动写入：
     - `opponent_confirm_deadline_at`
     - `timeout_minutes`
   - 超时后可把赛果从 `awaiting_opponent_confirmation` 升级为 `awaiting_admin_review`
   - 同时推送管理员处理卡片

3. **补 disputed → disputes 自动衔接**
   - 对手拒绝赛果时，自动在 `disputes` 建一条争议记录
   - 超时升级管理员时，也会自动生成关联争议记录
   - `matches.latest_dispute_uid` / `result_reports.linked_dispute_uid` 会尽量同步回写

### 2.2 report_status 建议枚举

- `awaiting_opponent_confirmation`
- `opponent_confirmed`
- `opponent_rejected`
- `awaiting_admin_review`
- `admin_confirmed`

### 2.3 opponent_confirmation_status 建议枚举

- `pending`
- `confirmed`
- `rejected`
- `timeout`
- `skipped`

## 5. disputes 自动衔接字段建议

当前自动创建争议时会尽量写入以下字段：

| 字段名 | 说明 |
|---|---|
| dispute_uid | 争议 UID |
| tournament_uid | 赛事 UID |
| match_uid | 对局 UID |
| status | 初始写 `admin_review` |
| dispute_type | 建议值：`赛果异议` / `赛果确认超时` |
| source_type | `result_report_rejected` / `result_report_timeout` |
| opponent_name_snapshot | 对手名称快照 |
| description | 自动拼接当前赛果与拒绝/超时原因 |
| requested_resolution | 默认请管理员核实最终赛果 |
| occurred_at | 争议触发时间 |
| created_at | 创建时间 |
| plaintiff_player_uid | 发起异议方 |
| defendant_player_uid | 被异议方 |
| related_result_report_uid | 关联赛果上报 UID |
| evidence_summary | 赛果上报备注 |

## 6. 环境变量

当前仍使用以下环境变量接真实表：

- `HS_RESULT_REPORTS_TABLE_ID`
- `HS_MATCH_GAMES_TABLE_ID`
- `HS_MATCH_CLASS_STATES_TABLE_ID`
- `HS_RESULT_REPORT_CONFIRM_TIMEOUT_MINUTES`（可选，默认 30）

---

## 7. 下一步最直接落地顺序

建议按这个顺序继续：

1. 先建 `result_reports`
2. 再建 `match_games`
3. 再建 `match_class_states`
4. 把 3 个 table_id 配入环境变量
5. 找一场测试赛跑一遍：
   - submit report
   - opponent confirm / reject
   - timeout escalate
   - admin confirm

这样就能把个人赛 BO5 从“状态可算”推进到“线上留痕可审计”。
