# 回调映射与写回规则 v1

> 炉石赛事自动化系统 - 按钮回调 Action 与 Bitable 写回映射
> 版本: v1.0 | 日期: 2026-03-28

---

## 1. 设计原则

### 1.1 回调处理流程
```
用户点击按钮
    ↓
飞书推送 callback 事件
    ↓
机器人解析 action + 参数
    ↓
执行写回逻辑（可能多表）
    ↓
更新原卡片状态（或推送新卡片）
    ↓
发送通知给相关人员
```

### 1.2 写回策略
- **原子性**：单次回调涉及的多表更新视为一个事务
- **幂等性**：同一回调重复触发不会导致重复写回
- **可追溯**：所有写回操作记录到 `operation_logs`（建议后续建表）
- **失败重试**：写回失败时自动重试3次，仍失败则通知管理员

## 2. Action 命名规范

### 2.1 命名格式
```
{verb}_{resource}_{operation}
```

### 2.2 动词列表
| 动词 | 含义 | 示例 |
|------|------|------|
| approve | 审核通过 | approve_registration |
| reject | 审核驳回 | reject_deck_submission |
| request | 要求补充/请求操作 | request_registration_info |
| view | 查看详情 | view_deck_detail |
| refresh | 刷新状态 | refresh_status |
| list | 列出待办 | list_pending_registrations |
| publish | 发布/公示 | publish_decks |
| lock | 锁定 | lock_registration |

### 2.3 资源列表
- `registration` - 报名记录
- `deck_submission` - 卡组提交
- `dispute` - 争议
- `match` - 对局
- `announcement` - 公告

## 3. 核心 Action 映射表

### 3.1 报名审核相关

#### Action: approve_registration
**触发条件**：管理员点击报名审核卡片的「通过」按钮

**参数**
```json
{
  "action": "approve_registration",
  "registration_id": "recxxx",
  "entity_type": "player|team",
  "entity_uid": "player_xxx|team_xxx",
  "admin_open_id": "ou_xxx",
  "timestamp": "2026-03-28T12:00:00+08:00"
}
```

**写回操作**

| 顺序 | 目标表 | 操作 | 字段更新 |
|------|--------|------|----------|
| 1 | registrations | UPDATE | registration_status = "approved", review_status = "approved", reviewed_by = admin_open_id, reviewed_at = timestamp |
| 2 | players/teams | UPDATE（可选） | verified_status = "verified" |

**后续动作**
- 更新原卡片：显示「已通过」状态，隐藏操作按钮
- 发送通知给选手/队长：「您的报名已通过审核」
- 如为个人赛：推送卡组提交入口卡片

---

#### Action: reject_registration
**触发条件**：管理员点击「驳回」按钮

**参数**
```json
{
  "action": "reject_registration",
  "registration_id": "recxxx",
  "reject_reason": "资料不完整，请补充...",
  "admin_open_id": "ou_xxx",
  "timestamp": "2026-03-28T12:00:00+08:00"
}
```

**写回操作**

| 顺序 | 目标表 | 操作 | 字段更新 |
|------|--------|------|----------|
| 1 | registrations | UPDATE | registration_status = "rejected", review_status = "rejected", review_note = reject_reason, reviewed_by = admin_open_id, reviewed_at = timestamp |

**后续动作**
- 更新原卡片：显示「已驳回」+ 驳回原因
- 发送通知给选手/队长：「您的报名被驳回，原因：...」
- 提供「修改重提」按钮

#### Action: request_registration_info
**触发条件**：管理员点击「要求补充」按钮

**参数**
```json
{
  "action": "request_registration_info",

  "request_note": "请补充定妆照",
```

**写回操作**

**触发条件**：管理员点击卡组审核卡片的「通过」按钮

**参数**
```json
{
  "action": "approve_deck_submission",
  "submission_id": "recxxx",
  "registration_id": "recxxx",
  "entity_type": "player|team",
  "admin_open_id": "ou_xxx",
```

**写回操作**

| 顺序 | 目标表 | 操作 | 字段更新 |
|------|--------|------|----------|
| 1 | deck_submissions | UPDATE | submission_status = "approved", reviewed_by = admin_open_id, reviewed_at = timestamp, is_current_effective_version = true |
| 2 | deck_submissions | UPDATE（条件） | 如存在旧版本：previous_submission_uid 指向旧版本，旧版本 is_current_effective_version = false, submission_status = "superseded" |

**后续动作**
- 更新原卡片：显示「已通过」
- 通知提交人：「您的卡组已通过审核」

### 3.4 工作台查询相关

#### Action: list_pending_registrations
**触发条件**：管理员点击工作台的「处理报名」按钮

**处理逻辑**
1. 查询 registrations 表：
   ```
   filter: registration_status = "submitted" OR review_status = "pending"
   sort: submitted_at desc
   limit: 10
   ```
2. 为每条记录生成报名审核卡片
3. 分批推送给管理员（如数量多）

---

#### Action: list_pending_decks
**处理逻辑**
1. 查询 deck_submissions 表：
   ```
   filter: submission_status = "submitted"
   ```
2. 生成卡组审核卡片列表

---

#### Action: list_pending_disputes
**处理逻辑**
1. 查询 disputes 表：
   ```
   filter: status = "admin_review"
   ```
2. 生成争议裁决卡片列表

## 5. 错误处理与重试机制

### 5.1 写回失败处理

```javascript
async function executeWriteback(operations) {
  const maxRetries = 3;
  
  for (const op of operations) {
    let success = false;
    let attempts = 0;
    
    while (!success && attempts < maxRetries) {
      try {
        await executeOperation(op);
        success = true;
      } catch (error) {
        attempts++;
        if (attempts >= maxRetries) {
          // 记录失败日志
          await logFailedOperation(op, error);
          // 通知管理员
          await notifyAdmin(`写回失败: ${op.table} ${op.action}`);
          throw error;
        }
        // 延迟重试
        await delay(1000 * attempts);
      }
    }
  }
}
```

### 5.2 幂等性保证
- 使用 `registration_id` + `action` + `admin_open_id` + `timestamp` 作为唯一键
- 写回前检查是否已处理过相同请求

## 6. 回调安全校验

### 6.1 必校验项
1. **来源校验**：确认 callback 来自飞书官方
2. **权限校验**：确认 admin_open_id 有权限操作该记录
3. **状态校验**：确认记录当前状态允许该操作
4. **时效校验**：确认 timestamp 在合理范围内（±5分钟）

### 6.2 权限检查逻辑
```javascript
async function checkPermission(adminOpenId, tournamentUid, action) {
  // 查询 tournament_admins 表
  const admin = await queryTournamentAdmin(tournamentUid, adminOpenId);
  
  if (!admin) {
    throw new Error('无权限操作该赛事');
  }
  
  // 检查 action 是否在允许列表
  const allowedActions = getAllowedActions(admin.role);
  if (!allowedActions.includes(action)) {
    throw new Error('当前角色无权执行此操作');
  }
  
  return true;
}
```
