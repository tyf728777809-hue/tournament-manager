# 个人赛赛果线上回调接线方案 v1

> 日期：2026-03-29
> 目标：把当前已完成真实闭环验证的个人赛赛果链路，接到飞书线上卡片回调入口上。

---

## 1. 本次接线范围

本次不是重写业务逻辑，而是把现有 `callback-handler.js` 真正挂到一个可接飞书事件的 HTTP webhook 上。

本轮最小上线范围：
1. 接收飞书 `url_verification`
2. 接收飞书交互式卡片按钮回调
3. 将回调里的 `action.value` + `operator.open_id` 适配为：
   - `callback`
   - `context.operatorOpenId`
4. 调用 `handleCallback()` 执行业务写回与通知

---

## 2. 本轮新增文件

- `炉石赛事/feishu-callback-adapter.js`
  - 负责解析飞书事件 envelope
  - 支持未加密 payload
  - 若配置 `FEISHU_ENCRYPT_KEY`，支持解密 `encrypt` 信封
  - 统一提取 `action.value` / `operatorOpenId`

- `炉石赛事/feishu-callback-server.js`
  - 最小 HTTP 服务入口
  - 默认监听：`POST /feishu/callback`
  - 健康检查：`GET /healthz`

---

## 3. 环境变量

至少建议配置：

```bash
export FEISHU_ACCESS_TOKEN=xxx
export FEISHU_VERIFICATION_TOKEN=xxx
export FEISHU_CALLBACK_PORT=8787
export FEISHU_CALLBACK_PATH=/feishu/callback
```

如飞书事件订阅开启了加密，再补：

```bash
export FEISHU_ENCRYPT_KEY=xxx
```

说明：
- `FEISHU_ACCESS_TOKEN`：当前代码写表/发消息所需
- `FEISHU_VERIFICATION_TOKEN`：用于校验回调 token
- `FEISHU_ENCRYPT_KEY`：仅在飞书启用 encrypt 信封时需要

---

## 4. 启动方式

### 4.1 正常模式

```bash
node 炉石赛事/feishu-callback-server.js
```

### 4.2 本地 dry-run 模式（推荐先用这个联调入口）

```bash
FEISHU_CALLBACK_DRY_RUN=1 \
node 炉石赛事/feishu-callback-server.js
```

这个模式下：
- 会正常接收 HTTP 请求
- 会正常解析飞书事件 / action.value / operatorOpenId
- **不会执行** `handleCallback()` 里的写表和通知
- 适合先验证“入口是否通、事件是否解对、字段是否抽对”

默认会监听：

```text
http://0.0.0.0:8787/feishu/callback
```

健康检查：

```bash
curl http://127.0.0.1:8787/healthz
```

## 5. 飞书后台接线要点

在飞书开放平台里，把事件订阅 / 卡片回调地址指向：

```text
https://<your-domain>/feishu/callback
```

至少要完成两步验证：
1. `url_verification` 能返回 `challenge`
2. 点击卡片按钮后，服务端能收到事件并进入 `handleCallback()`

## 6. 当前已修正的一个关键一致性点

### admin-confirm 状态保持与真实闭环一致

在 `callback-handler.js` 中，`adminConfirmMatchResultReport()` 已调整为：
- **优先保留原 report 上已有的 `opponent_confirmation_status`**
- 仅在完全缺失时，才回退为：
  - `awaiting_admin_review` → `timeout`
  - 其他情况 → `skipped`

这样就不会再把这轮真实样本中已验证的：

```text
submit -> timeout -> admin-confirm
=> opponent_confirmation_status = timeout
```

错误覆盖成 `skipped`。

## 8. 建议联调顺序

1. 本地以 dry-run 模式启动 `feishu-callback-server.js`
2. 用 `url_verification` 验证回调地址可用
3. 用 `confirm_match_result_report` 样板请求验证：
   - `action` 是否抽对
   - `match_uid / result_report_uid` 是否抽对
   - `operatorOpenId` 是否抽对
4. 再切到真实执行模式联调：
   - `reject_match_result_report`
   - `escalate_match_result_report_timeout`
   - `admin_confirm_match_result_report`
5. 确认 timeout 后 admin-confirm 仍保留 `opponent_confirmation_status = timeout`

### 8.1 本地 dry-run 样板

仓库已补两个最小样板：
- `tmp/feishu-url-verification.json`
- `tmp/feishu-interactive-confirm.json`

也已补一个一键本地脚本：
- `炉石赛事/run-feishu-callback-local.sh`

本地可直接这样验证：

```bash
curl -X POST http://127.0.0.1:8787/feishu/callback \
  -H 'Content-Type: application/json' \
  --data @tmp/feishu-url-verification.json
```

```bash
curl -X POST http://127.0.0.1:8787/feishu/callback \
  -H 'Content-Type: application/json' \
  --data @tmp/feishu-interactive-confirm.json
```

或者直接用一键脚本：

```bash
FEISHU_VERIFICATION_TOKEN=test-verification-token \
炉石赛事/run-feishu-callback-local.sh dry-run tmp/feishu-interactive-confirm.json
```

当 `FEISHU_ACCESS_TOKEN` 就绪后，可切真实执行模式：

```bash
FEISHU_VERIFICATION_TOKEN=xxx \
FEISHU_ACCESS_TOKEN=xxx \
炉石赛事/run-feishu-callback-local.sh real tmp/feishu-interactive-confirm.json
```
