# 个人赛赛果用户身份写表方案 v1

> 日期：2026-03-29

## 1. 背景

在当前环境中，飞书应用 `tenant_access_token` 对运营中心 Bitable App：
- **可读**
- **不可写**（`91403 Forbidden`）

但使用用户身份工具对同一批表：
- `matches`
- `result_reports`
- `disputes`

可以正常写入。

因此赛果链路改为：

1. `callback-handler.js` 继续负责业务逻辑、权限校验、状态机推进
2. 写表阶段不再强耦合应用身份 API
3. 新增 `user_identity_plan` 模式：
   - 先产出 `execution.write_plan`
   - 再由当前会话用用户身份工具执行计划

---

## 2. 本轮改动

### 2.1 callback-handler.js
新增执行上下文：
- `beginExecutionContext()`
- `withExecutionMeta()`
- `writeMode = api | user_identity_plan`

新增行为：
- `safeCreateRecord / safeUpdateRecord / safeUpsertByUid`
  - 在 `api` 模式下：照常直接调用应用身份 API
  - 在 `user_identity_plan` 模式下：不直接写表，改为输出写入计划

### 2.2 hs-solo-result-smoke.js
新增参数：
- `--write-mode user_identity_plan`

用于把赛果链路跑成“可执行计划”，便于当前会话直接接管写表。

## 5. 当前意义

这意味着：

- **tenant token 写表不再是主阻塞项**
- 个人赛赛果链路已经可以在用户身份模式下继续推进
- 后续可以继续把：
  - timeout escalate
  - confirm 分支
  - BO5 小局自动推进
  - 争议裁决闭环

都统一到这套“业务逻辑 + 用户身份执行器”结构里。
