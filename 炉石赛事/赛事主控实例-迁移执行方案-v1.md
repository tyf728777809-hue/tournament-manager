# 赛事主控实例迁移执行方案 v1

> 目标：把炉石赛事系统从“个人助理主导”迁移为“独立赛事主控 agent 主导”，并使用独立飞书机器人承接赛事入口。

---

## 1. 迁移目标

### 要达成
- 个人助理继续服务用户本人
- 新赛事主控实例接管赛事系统运营盘
- 新飞书 bot 承接赛事群与赛事管理员私聊入口
- 飞书多维表格 + 飞书文档继续作为共享真相源

### 不要做成
- 在当前个人助理实例里继续混合承接赛事主控
- 让赛事主控继承私人长期记忆
- 一次性硬切所有入口

---

## 2. 推荐架构

### 实例 A：个人助理实例
- 保留当前职责
- 保留当前私人入口
- 不再作为赛事默认主控

### 实例 B：赛事主控实例
- 独立 OpenClaw
- 独立 workspace
- 独立飞书 bot
- 独立运行日志、记忆、SOP
- 只处理赛事相关事务

### 共享层
- 飞书多维表格
- 飞书文档
- 明确的 UID / 结构化数据

---

## 3. 实施阶段

### Phase 1：准备
- 建立赛事主控实例工作区
- 放入 core files
- 准备新飞书 bot
- 准备新实例配置

### Phase 2：接线
- 给新实例配置飞书渠道
- 配置新 bot 的回调 URL
- 打通 url_verification
- 验证最小消息与回调链路

### Phase 3：灰度
- 先接一个测试赛事
- 先接一个测试群
- 先接一个管理员入口
- 验证真实写表与异常兜底

### Phase 4：正式切流
- 新赛事默认走新主控
- 旧流程保留观察期
- 出现异常时可快速回滚

---

## 4. 最小必备物料

迁移前，赛事主控实例至少应具备：
- `SOUL.md`
- `AGENTS.md`
- `USER.md`
- `BOOTSTRAP.md`
- agent core files
- 赛事文档索引
- 迁移 checklist
- 回滚方案

---

## 5. 成功标准

迁移成功的标志不是“新实例能回复消息”，而是：
- 能独立承接赛事入口
- 能稳定推进关键赛事流程
- 不依赖私人助理上下文
- 不出现跨赛事串线
- 有清晰回滚路径

---

## 6. 当前建议动作

### 现在先做
1. 完成赛事主控实例骨架文件
2. 创建新实例工作区
3. 创建新飞书 bot
4. 在新实例做最小接线验证

### 暂时不要做
1. 不要替换现有个人助理入口
2. 不要直接把全部正式赛事切过去
3. 不要把私人记忆复制到新实例
4. 不要在未灰度时直接宣布正式接管

---

## 7. 一句话执行策略

> 独立实例、独立 bot、共享数据、渐进接管、保留回滚。
