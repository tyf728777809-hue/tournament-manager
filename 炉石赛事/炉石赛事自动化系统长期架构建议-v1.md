# 炉石赛事自动化系统长期架构建议 v1

更新时间：2026-03-29

---

## 1. 结论先行

长期目标如果是：
- 稳定好用
- 可以真正实现
- 后续能持续扩展而不是反复返工

建议采用：

**OpenClaw（智能与运营层） + 独立业务交互服务（确定性业务层） + Feishu Bitable（结构化真相层）**

一句话概括：

- **OpenClaw 负责“会话智能、提醒、解释、编排、管理辅助”**
- **独立业务服务负责“按钮回调、表单提交、状态机推进、业务写表”**
- **Bitable 负责“结构化数据落地与审计”**

这意味着：

> 不再把 OpenClaw 当前 Feishu 卡片回调机制当作整个赛事系统的主业务入口。

---

## 2. 为什么不建议继续把 OpenClaw 卡片机制作为主干

当前排查结论已经足够说明问题：

1. OpenClaw Feishu 插件当前没有为自定义业务 action 提供通用 fallback 路由
2. 某些卡片按钮点击事件到插件层时，`action.value` 已经变成 `{}`
3. `ocf1` envelope 更像内部交互协议，不适合作为你长期业务协议的基础
4. 就算补丁打通，后续升级仍可能反复适配

因此它适合：
- 辅助型交互
- 内部实验
- 非关键流程

不适合：
- 报名主入口
- 卡组提交主入口
- 赛果确认核心动作
- 争议提交与裁决闭环

---

## 3. 推荐的长期分层架构

### 3.1 三层结构

#### A. 智能与运营层：OpenClaw
负责：
- 管理员对话
- FAQ
- 状态查询
- 提醒与催办
- 审核建议
- 争议 AI 辅助分析
- 群公告草拟
- 多赛事流程编排

特点：
- 强在理解上下文、生成文本、调工具
- 不承担关键按钮回调的确定性执行

#### B. 业务交互层：hs-esports-service（建议独立）
负责：
- Feishu 卡片回调
- H5 页面表单
- 飞书原生表单入口接线
- 业务状态机推进
- 权限校验
- 幂等控制
- 写 Bitable
- 发业务通知

特点：
- 逻辑清晰、可测试、可追踪
- 不依赖 OpenClaw 内部卡片实现细节

#### C. 结构化真相层：Feishu Bitable / Docs
负责：
- 基础资料
- 报名记录
- 队员名单
- 卡组版本
- BP 流程
- 比赛结果
- 争议记录
- 公告与提醒记录

特点：
- 作为唯一结构化数据真相来源
- OpenClaw 与业务服务统一读写

## 4. 各能力建议放哪一层

### 4.1 报名

**建议主承载：业务交互层 + 表单/H5**

原因：
- 字段多
- 需要修改
- 有资料复用
- 有身份匹配
- 后续要审核

推荐实现：
- 群里放“报名入口卡片”
- 点击后打开 H5 或飞书表单
- 提交后业务服务写 `players / teams / registrations / registration_members`
- OpenClaw 负责后续解释、提醒、审核辅助

### 4.2 卡组提交

**建议主承载：业务交互层 + H5 页面**

原因：
- 字段结构复杂
- 需要职业校验
- 可能有图片/代码
- 需要版本管理
- 可能要补丁重交

推荐实现：
- 报名审核通过后，由业务服务发“卡组提交入口”
- 入口打开独立页面
- 页面完成校验与保存
- 写 `deck_submissions / decks`
- OpenClaw 负责审核说明、异常解释、通知

### 4.3 赛果确认

**建议主承载：业务交互层 + Feishu 卡片回调**

这是最适合保留“卡片按钮确认”体验的模块。

原因：
- 交互短
- 状态机明确
- 需要即时确认
- 按钮价值最高

推荐实现：
- 业务服务自己发送确认卡片
- 业务服务自己接回调
- 写 `result_reports / matches / disputes`
- OpenClaw 负责：
  - 查询当前状态
  - 给管理员解释异常
  - 提供争议裁决建议

### 4.4 争议提交

**建议主承载：业务交互层 + 表单/H5**

原因：
- 需要长文本
- 需要附件
- 可能反复补充证据
- 表单式最自然

### 4.5 管理员审核与运营

**建议主承载：OpenClaw + Bitable 视图 + 少量业务卡片**

最适合交给 OpenClaw 的部分：
- “帮我列出待审报名”
- “帮我看这场争议的规则依据”
- “帮我生成公告”
- “谁还没交卡组”
- “帮我发提醒”

## 5. 推荐的产品边界

### 5.1 OpenClaw 的职责边界

OpenClaw 应负责：
- 面向管理员的对话入口
- 面向群的说明、提醒、答疑
- 对结构化数据做查询与解释
- 对流程节点做编排与建议
- 对争议做 AI 辅助判断

OpenClaw 不应负责：
- 核心业务按钮的唯一回调入口
- 大型表单采集主链路
- 复杂前端交互
- 高确定性的长状态机前端

### 5.2 业务服务的职责边界

业务服务应负责：
- Feishu 卡片按钮回调
- 表单提交
- 身份映射
- 幂等与状态机
- Bitable 写入
- 业务通知发送
- 审计日志

---

## 6. 推荐的交互策略

### 6.1 群聊入口策略

群里保留卡片，但卡片职责要收敛：
- 进入报名
- 查看规则
- 打开提交入口
- 确认赛果
- 查看当前状态

不要让群卡片承担：
- 长表单采集
- 多步复杂编辑
- 大量输入

### 6.2 私聊策略

私聊里适合：
- 提醒
- 审核结果通知
- 赛果确认
- 补材料提示
- FAQ
- 状态解释

### 6.3 表单与页面策略

- **短动作**：卡片按钮回调
- **长输入**：表单 / H5
- **复杂编辑**：H5
- **管理员工作台**：Bitable 视图 + OpenClaw 查询

## 7. 技术落地建议

### 7.1 新建独立业务服务（建议）

建议新增一个明确边界的服务：

- 名称建议：`hs-esports-service`
- 目录建议：当前 hs-esports 工作区内单独模块化
- 职责：
  - callback server
  - card builder
  - form/H5 server
  - tournament workflow engine
  - bitable adapter

### 7.2 模块划分建议

建议拆为以下模块：

1. `identity/`
   - open_id 映射
   - 玩家 / 队长 / 管理员识别

2. `workflows/`
   - signup workflow
   - deck submission workflow
   - result report workflow
   - dispute workflow

3. `cards/`
   - card builder
   - callback parser
   - card action router

4. `forms/` 或 `pages/`
   - signup page
   - deck submission page
   - dispute page

5. `storage/`
   - bitable read/write adapter
   - document helper

6. `notifier/`
   - 私聊提醒
   - 群消息
   - 管理员通知

### 7.3 协议建议

不要再以 OpenClaw 内部 envelope 作为业务协议。

建议业务层自己定义稳定协议，例如：

```json
{
  "biz": "hs_esports",
  "action": "confirm_result_report",
  "tournament_uid": "...",
  "match_uid": "...",
  "result_report_uid": "...",
  "version": "v1"
}
```

即便最终仍使用 Feishu 卡片，也应由**你的业务服务自己定义和解析**。

## 8. 三期实施路线图

### 第一期：先把“可稳定上线”的主链路跑通

目标：
- 报名能稳定收集
- 卡组能稳定提交
- OpenClaw 能查、能提醒、能解释

建议范围：
1. 报名入口改为表单/H5
2. 卡组提交改为表单/H5
3. 争议提交改为表单/H5
4. OpenClaw 保留查询、FAQ、提醒、审核辅助

第一期不强求：
- 赛果确认卡片闭环
- 复杂卡片回调

### 第二期：把高价值即时动作独立出来

目标：
- 赛果确认卡片真正可用
- 管理员审批卡片真正可用

建议范围：
1. 业务服务自己发送确认卡片
2. 业务服务自己接回调
3. 实现幂等和状态机校验
4. OpenClaw 只做辅助解释和异常处置

### 第三期：形成完整赛事运营中枢

目标：
- OpenClaw 成为管理员总控台
- 业务服务成为稳定执行引擎

建议范围：
1. 多赛事并行
2. 统一工作台
3. 自动提醒排程
4. 争议 AI 辅助闭环
5. 公告与运营 SOP 自动化

## 9. 对当前代码的直接影响

### 9.1 可以继续保留的部分

当前这些代码仍有价值：
- `callback-handler.js` 内的业务规则
- `notification-utils.js` 内的通知拼装能力
- `bitable-api-utils.js` 的数据访问能力
- 各类设计稿与状态机文档

### 9.2 需要调整的方向

需要逐步把当前代码从：

> “依赖 OpenClaw 卡片回调触发业务”

调整为：

> “业务服务自己接业务回调，OpenClaw 通过工具/查询/提醒协同”

### 9.3 建议优先抽离的模块

优先抽离：
1. 报名入口与表单处理
2. 卡组提交入口与校验
3. 赛果确认回调
4. 争议提交

---

## 10. 关键决策建议

### 建议立刻确认的决策

#### 决策 1
**OpenClaw 不再作为核心业务卡片回调主入口。**

#### 决策 2
**报名、卡组、争议等结构化输入优先迁移到表单/H5。**

#### 决策 3
**赛果确认保留卡片体验，但回调交给独立业务服务。**

#### 决策 4
**OpenClaw 作为赛事运营中枢，而不是业务前端框架。**

## 12. 最终建议（简版）

长期来看，最稳、最能真正实现的路线是：

**把结构化采集与确定性按钮回调独立出去，把 OpenClaw 保留为智能中枢与运营助手。**

这条路线的优点是：
- 技术边界清晰
- 升级风险可控
- 可测试、可审计
- 适合逐步演进成真正可运营的赛事系统

而不是停留在“聊天机器人 + 一堆临时卡片 hack”的状态。
